# 分布式 ID 生成器竞品对比分析报告

> **对比对象**：mysql-sequence vs 美团 Leaf vs 百度 UidGenerator vs 滴滴 Tinyid
>
> **报告日期**：2026-06-06
>
> **报告目的**：通过系统性对比，明确 mysql-sequence 的差异化定位、竞争优势与改进方向

### 定位差异一句话总结

- **Leaf**：通用分布式 ID 服务，号段 + 雪花双模式，独立部署
- **UidGenerator**：高性能 Snowflake 变体，RingBuffer 缓存，单机极限吞吐
- **Tinyid**：Leaf 号段模式的简化实现，多数据源 + 客户端 SDK
- **mysql-sequence**：Oracle 序列语义兼容的嵌入式 Starter，号段 + 严格连续双模式

## 二、核心架构对比

### 2.1 算法模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                        算法模型对比                                   │
├──────────────┬──────────────┬──────────────┬──────────────┬─────────┤
│              │ mysql-seq    │ Leaf          │ UidGenerator │ Tinyid  │
├──────────────┼──────────────┼──────────────┼──────────────┼─────────┤
│ 号段模式      │ ✅ CACHED    │ ✅ segment   │ ❌           │ ✅      │
│ 雪花模式      │ ❌          │ ✅ snowflake │ ✅ 变体      │ ❌      │
│ 严格连续      │ ✅ STRICT    │ ❌           │ ❌           │ ❌      │
│ 双 Buffer     │ ✅          │ ✅           │ ❌ RingBuf   │ ✅      │
│ 动态步长      │ ✅ 阻尼调整  │ ❌           │ ❌           │ ❌      │
│ Oracle 语义   │ ✅          │ ❌           │ ❌           │ ❌      │
└──────────────┴──────────────┴──────────────┴──────────────┴─────────┘
```

**关键差异**：

1. **STRICT 模式**是 mysql-sequence 独有的。其他三个项目均不提供 `SELECT ... FOR UPDATE` 严格连续保证，无法满足财务流水号等"绝不跳号"的场景
2. **Oracle 序列语义**（CURRVAL/NEXTVAL/CYCLE/INCREMENT BY/START WITH）仅 mysql-sequence 支持，其他项目均为纯 ID 生成器，不关心 Oracle 兼容性
3. **动态步长调整**仅 mysql-sequence 实现。Leaf 提出了线性公式但未在开源版本落地；mysql-sequence 参考该公式并加入了阻尼控制（单次调整不超过 2 倍）

### 2.3 数据模型

| 项目 | 核心表 | 说明 |
|---|---|---|
| **mysql-sequence** | `sequence_config` + `sequence_record` | 配置表 + 审计记录表，2 张表 |
| **Leaf** | `leaf_alloc` | 仅 1 张号段分配表 |
| **UidGenerator** | `WORKER_NODE` | 仅 WorkerId 分配表 |
| **Tinyid** | `tiny_id_info` + `tiny_id_token` | 号段表 + 鉴权 Token 表 |

mysql-sequence 的 `sequence_record` 表是独有的审计追踪设计，记录每个号段的分配历史、实例归属和生命周期状态（ALLOCATED → EXPIRED → RECYCLED），其他项目均无此能力。

## 三、性能对比

### 3.1 吞吐量

| 模式 | mysql-sequence | Leaf | UidGenerator | Tinyid |
|---|---|---|---|---|
| 号段/缓存模式 | ≥ 5,000 TPS | ~50,000 TPS | — | ~10,000,000 TPS（官方声称） |
| Snowflake 模式 | — | ~409.6 万 TPS（理论） | ~6,000,000+ TPS | — |
| 严格连续模式 | ~50 TPS（单实例） | — | — | — |

### 3.2 延迟

| 指标 | mysql-sequence | Leaf | UidGenerator | Tinyid |
|---|---|---|---|---|
| TP999（CACHED） | ≤ 1ms | ≤ 1ms | < 1ms | 未公布 |
| 号段切换等待 | 零（双 Buffer） | 零（双 Buffer） | 零（RingBuffer） | 零（双 Buffer） |
| DB 宕机容忍 | ≤ 5 分钟 | 号段耗尽前可用 | 不依赖 DB（运行时） | 号段耗尽前可用 |

### 3.3 性能分析

- **mysql-sequence CACHED 模式的 5,000 TPS 相对保守**。Leaf 同为号段模式达到 50,000 TPS，差距主要来自：
  - mysql-sequence 的 CACHED 模式每次 `nextVal` 都经过 `SequenceService` 路由 + `SessionHolder` 写入 + 日志记录，链路更长
  - Leaf 直接从 `SegmentBuffer` 的 `AtomicLong` 取值，链路极短
  - mysql-sequence 的 5,000 TPS 是**单实例保守估计**，实际压测可能更高

- **UidGenerator 的 600 万+ TPS 不具可比性**：Snowflake 算法是纯内存计算，无 DB 交互，与号段模式本质不同

- **Tinyid 的 1000 万+ TPS 为官方声称**，缺乏第三方验证

### 功能亮点解读

- **mysql-sequence 的 Oracle 语义矩阵**（CURRVAL + CYCLE + INCREMENT BY + START WITH + MIN/MAXVALUE）是其他项目完全不具备的，这是从 Oracle 迁移到 MySQL 场景的核心需求
- **STRICT 模式**填补了"严格连续序列"的市场空白，其他项目均不提供此能力
- **动态步长 + 阻尼控制**是 mysql-sequence 在 Leaf 基础上的重要工程化改进，防止步长振荡导致 DB 负载波动
- **配置热更新**（定时清空缓存 + 懒加载）是运维友好设计，其他项目均需重启

- **Tinyid 代码质量一般**，功能实现完整但缺乏深度优化

## 十、改进建议

### 高优先级

| # | 建议 | 依据 | 参考 |
|---|---|---|---|
| 1 | **CACHED 模式性能优化**：减少 nextVal 链路开销（SessionHolder 写入可延迟/异步），目标提升至 20,000+ TPS | 当前 5,000 TPS 与 Leaf 的 50,000 TPS 差距明显 | Leaf SegmentBuffer 直接 AtomicLong 取值 |

## 附录：竞品详细资料

### A. 美团 Leaf

- **仓库**：https://github.com/Meituan-Dianping/Leaf
- **设计文章**：https://tech.meituan.com/2017/04/21/mt-leaf.html
- **核心贡献**：号段双 Buffer 模式的原创设计，被 mysql-sequence 和 Tinyid 直接参考
- **ID 结构（Snowflake 模式）**：`sign(1) | timestamp(41) | workerId(5) | datacenterId(5) | sequence(12)`

### B. 百度 UidGenerator

- **仓库**：https://github.com/baidu/uid-generator
- **核心贡献**：RingBuffer 预生成 + CacheLine 消除伪共享 + 借用未来时间突破 Snowflake 单秒上限
- **ID 结构**：`sign(1) | deltaSeconds(28) | workerId(22) | sequence(13)`，各段位数可配置

### C. 滴滴 Tinyid

- **仓库**：https://github.com/didi/tinyid
- **核心贡献**：Leaf 号段模式的简化实现 + 多数据源支持 + 步长/余数模式（奇偶序列）
- **注意**：官方声明非滴滴正式产品
