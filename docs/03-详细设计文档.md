## 1. 文档概述

### 1.1 项目背景

mysql-sequence 是 Spring Boot Starter 组件，为 TDSQL/MySQL 环境提供 Oracle 兼容的 Sequence 能力。本项目打包为 jar，以库的形式嵌入宿主 Spring Boot 应用的上下文中，不是独立部署的服务。

Oracle 数据库提供原生 Sequence 对象（NEXTVAL / CURRVAL），而 MySQL / TDSQL 仅支持表级 AUTO_INCREMENT，无法满足多表共享序列、自定义步长/边界/循环策略、跨实例严格连续递增等需求。本项目填补这一空白。

### 1.2 功能范围

**包含功能：**
- 严格连续模式（STRICT）：基于 `SELECT ... FOR UPDATE` 悲观锁，多实例串行化
- 号段缓存模式（CACHED）：基于 Segment + 双 Buffer + 动态步长，高吞吐
- 原生序列模式（NATIVE）：基于 TDSQL 原生 `tdsql_nextval()`，消除应用层行锁
- `nextVal(seqName)`：统一入口，根据 mode 路由分发
- `currVal(seqName)`：基于 ThreadLocal 的会话级当前值查询（Oracle CURRVAL 语义）
- 双 Buffer 零等待切换：当前 Segment 消耗达到阈值时异步预加载备用 Segment
- 动态步长调整：根据消耗速率自动调整步长，使 DB 更新周期稳定在 15min 目标值
- 循环序列支持（CYCLE / NOCYCLE）
- 自定义步长（INCREMENT BY）、边界值（MINVALUE / MAXVALUE）、起始值（START WITH）
- 优雅关闭：监听 `ContextClosedEvent` + `@PreDestroy`，持久化未使用号段
- 持久化标记：加载号段时插入 `sequence_record` 记录，用于审计追踪
- 监控接口：`GET /sequence/{seqName}` 查询单序列状态，`GET /sequence/list` 查询所有序列
- Spring Boot Starter 自动配置：独立 DataSource、条件创建 Bean
- 配置缓存定时刷新：通过 `ScheduledExecutorService` 定期清空配置缓存，感知 DB 配置变更
- 原生序列启动同步：`NativeSequenceSynchronizer` 在启动时自动创建缺失的 DB Sequence
- Level-2 懒创建自愈：`NativeStrictSequence` 在运行时自动创建缺失的 DB Sequence 并重试

**不包含功能：**
- Snowflake 算法
- 认证授权（auth）、告警（alerting）
- 分库分表（sharding）、多租户（multi-tenancy）
- 限流（rate limiting）
- 独立部署进程

### 2.2 开发规范

- **命名规范**：遵循 Google Java Style Guide（包名全小写、类名 UpperCamelCase、方法名 lowerCamelCase、常量 UPPER_SNAKE_CASE）
- **强制规约**：遵循阿里巴巴 Java 开发手册

- **设计原则**：SOLID 原则（单一职责、开闭、里氏替换、接口隔离、依赖倒置）

- **线程安全**：优先使用 `AtomicLong` 无锁机制，必要时使用 `ReentrantLock` + `volatile`
- **事务方式**：使用 `TransactionTemplate` 编程式事务（解决自调用 `@Transactional` 失效问题），不使用声明式事务注解

## 4. 类设计

### 4.1 model 包：数据实体

#### 4.1.1 SequenceConfig

**类路径**：`com.ccb.jx.seq.model.SequenceConfig`

**职责**：映射 `sequence_config` 表的实体类，承载序列的完整定义信息（Oracle 语义的完整映射）。该对象在 SequenceService 中缓存于 `ConcurrentHashMap`，并作为参数传递给 SequenceStrategy 实现。

**设计要点**：
- 使用 `Long`/`Integer` 包装类型而非原始类型，防止 MyBatis 映射 NULL 时自动拆箱导致 NPE
- 所有数值字段的 getter 提供 null-safe 默认值
- 提供无参构造器 + setter（MyBatis 映射要求）+ Builder 静态工厂方法（业务代码创建）
- 使用 `Date` 类型（非 `LocalDateTime`），兼容 Java 8 + MyBatis
- `cycle` 字段为 `Integer`（0/1），getter 返回 `Integer`，null-safe 默认 0
- `equals`/`hashCode` 仅基于 `seqName`

**类定义**：

```java
package com.ccb.jx.seq.model;

import java.util.Objects;
import java.util.StringJoiner;

public class SequenceConfig {

    private String seqName;
    private Long maxId = 1L;
    private Integer step = 1000;

    private Long minValue = 1L;

    private Integer cycle = 0;
    private Mode mode = Mode.CACHED;

    private String description;
    private Date updateTime;
    private Date createTime;

    public SequenceConfig() {}

    public static Builder builder(String seqName) { return new Builder(seqName); }

    public static class Builder {

        private Mode mode = Mode.CACHED;
        private Long startWith = 1L;
        private String description;

        Builder(String seqName) { this.seqName = seqName; }
        public Builder maxId(Long maxId) { this.maxId = maxId; return this; }
        public Builder step(Integer step) { this.step = step; return this; }
        public Builder incrementBy(Integer incrementBy) { this.incrementBy = incrementBy; return this; }
        public Builder minValue(Long minValue) { this.minValue = minValue; return this; }
        public Builder maxValue(Long maxValue) { this.maxValue = maxValue; return this; }
        public Builder cycle(Integer cycle) { this.cycle = cycle; return this; }
        public Builder mode(Mode mode) { this.mode = mode; return this; }
        public Builder startWith(Long startWith) { this.startWith = startWith; return this; }

        public SequenceConfig build() { /* 设置所有字段并返回 */ }
    }

    // getter / setter（null-safe getter）
    public Long getMaxId() { if (maxId == null) { return 1L; } return maxId; }
    public Integer getStep() { if (step == null) { return 1000; } return step; }
```

**字段说明**：

| 字段 | 类型 | null-safe 默认 | 说明 |
|------|------|----------------|------|
| seqName | String | - | 序列名称，全局唯一标识 |
| maxId | Long | 1L | 当前已分配的最大号段终点。STRICT 模式下表示最后一个已分配值；CACHED 模式下表示已分配的号段终点 |
| step | Integer | 1000 | 号段步长，仅 CACHED 模式使用。代表每次从 DB 批量预取的号段数量 |

| minValue | Long | 1L | 最小值，对应 Oracle `MINVALUE`。循环模式下的回绕目标值 |
| maxValue | Long | Long.MAX_VALUE | 最大值，对应 Oracle `MAXVALUE` |

### 4.2 buffer 包：双 Buffer 号段管理

#### 4.2.1 Segment

**类路径**：`com.ccb.jx.seq.buffer.Segment`

**职责**：表示从数据库预取的一段连续序列号范围 `[start, end]`。使用 `AtomicLong` 提供线程安全的号段分配，支持无锁并发。

**设计要点**：
- 双参数构造器 `Segment(long start, long end)` — 正常使用
- 三参数构造器 `Segment(long start, long end, boolean skipValidation)` — 供 `EmptySegment` 子类使用，跳过校验和初始化
- 使用 `getAndIncrement()` 而非 `getAndAdd(incrementBy)`，因为 CACHED 模式下 incrementBy 固定为 1
- `next()` 返回 -1 表示号段耗尽
- 包含 `initialized`（volatile boolean）和 `loadTime`（volatile long）字段，供 DoubleBuffer 判断号段状态
- `current` 字段在 `skipValidation=true` 时为 null（EmptySegment 场景）

```java
package com.ccb.jx.seq.buffer;

import java.util.concurrent.atomic.AtomicLong;

public class Segment {

    private final long start;
    private final long end;
    private final AtomicLong current;
    private volatile boolean initialized;
    private volatile long loadTime;

    public Segment(long start, long end) {
        if (start > end) { throw new IllegalArgumentException(...); }
        this.start = start;
        this.end = end;
        this.current = new AtomicLong(start);
        this.initialized = true;
        this.loadTime = System.currentTimeMillis();
    }

    protected Segment(long start, long end, boolean skipValidation) {
        this.start = start;
        this.end = end;
        if (!skipValidation) {
            if (start > end) { throw new IllegalArgumentException(...); }
            this.current = new AtomicLong(start);
            this.initialized = true;
            this.loadTime = System.currentTimeMillis();
        } else {
            this.current = null;
            this.initialized = false;
            this.loadTime = 0;
        }
    }

    public long next() {
        long value = current.getAndIncrement();
        if (value > end) { return -1; }
        return value;
    }
    public long getCurrent() { return current.get(); }
    public long remaining() { long cur = current.get(); if (cur > end) { return 0; } return end - cur + 1; }
    public long used() { return current.get() - start; }
    public boolean isExhausted() { return current.get() > end; }
    public double usage() { long total = end - start + 1; if (total <= 0) { return 1.0; } return Math.min(1.0, (double)(current.get() - start) / total); }
    public long getStart() { return start; }
    public long getEnd() { return end; }
    public boolean isInitialized() { return initialized; }
    public long getLoadTime() { return loadTime; }
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| start | long（final） | 号段起始值，包含 |
| end | long（final） | 号段结束值，包含 |
| current | AtomicLong | 当前分配位置，CAS 无锁递增。EmptySegment 中为 null |
| initialized | volatile boolean | 是否已初始化 |
| loadTime | volatile long | 号段加载时间戳（毫秒），用于动态步长计算 |

**方法说明**：

| 方法 | 返回值 | 说明 |
|------|--------|------|
| next() | long | 分配下一个值。使用 `getAndIncrement` 无锁操作。超出 end 返回 -1 |
| getCurrent() | long | 获取当前分配位置（不递增） |
| remaining() | long | 剩余可用数量 = `end - current + 1`，永不返回负数 |
| used() | long | 已使用数量 = `current - start` |
| isExhausted() | boolean | 判断是否耗尽：`current.get() > end` |
| usage() | double | 使用率，范围 [0.0, 1.0]，用于触发预加载判断 |
| isInitialized() | boolean | 号段是否已初始化 |
| getLoadTime() | long | 获取号段加载时间戳 |

#### 4.2.2 EmptySegment

**类路径**：`com.ccb.jx.seq.buffer.EmptySegment`

**职责**：Null Object 模式，表示尚未加载号段的状态。所有操作均返回"已耗尽"语义，用于替代 `null` 引用，消除 DoubleBuffer 中的 null 检查。

```java
package com.ccb.jx.seq.buffer;

public class EmptySegment extends Segment {

    public static final EmptySegment INSTANCE = new EmptySegment();

    private EmptySegment() {
        super(0, -1, true);
    }

    @Override public long next() { return -1; }
    @Override public long getCurrent() { return -1; }
    @Override public long remaining() { return 0; }
    @Override public long used() { return 0; }
    @Override public boolean isExhausted() { return true; }
    @Override public double usage() { return 1.0; }
    @Override public boolean isInitialized() { return false; }
    @Override public long getLoadTime() { return 0; }
    @Override public String toString() { return "EmptySegment"; }
}
```

**方法语义**：

| 方法 | 返回值 | 语义 |
|------|--------|------|
| next() | -1 | 号段耗尽 |
| remaining() | 0 | 无剩余 |
| isExhausted() | true | 已耗尽 |
| usage() | 1.0 | 使用率 100% |
| isInitialized() | false | 未初始化 |

#### 4.2.3 DoubleBuffer

**类路径**：`com.ccb.jx.seq.buffer.DoubleBuffer`

**职责**：管理两个 Segment 实例（current / standby），实现零等待切换。当前 Segment 消耗达到阈值时，触发异步预加载备用 Segment。

**设计要点**：
- 使用 `current` / `standby` 命名，初始值为 `EmptySegment.INSTANCE`（非 null）
- `preloadTriggered` 使用 `AtomicBoolean`（非 volatile boolean），保证 CAS 操作的原子性
- `staleResetCount`（AtomicInteger）监控异步任务被线程池丢弃后的活循环风险
- `asyncTaskSubmitter`（volatile Runnable）由外部 CachedSequence 设置，将 `loadStandby()` 提交到线程池
- 延迟初始化：构造时 `current = EmptySegment.INSTANCE`，首次 `next()` 时同步加载
- 双重条件预加载：使用率 >= 90% **或** 剩余数量 < max(50, 总大小 x 0.3)
- `MAX_RETRY = 3`：`next()` 方法最多重试 3 次
- 使用 `instanceof EmptySegment` 替代 null 检查

```java
package com.ccb.jx.seq.buffer;

import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import java.util.concurrent.atomic.AtomicBoolean;
import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.locks.ReentrantLock;
import java.util.function.Supplier;

public class DoubleBuffer {

    private static final double PRELOAD_THRESHOLD = 0.1;
    private static final int MAX_RETRY = 3;

    private volatile Segment current;
    private volatile Segment standby;
    private final ReentrantLock switchLock = new ReentrantLock();
    private final Supplier<Segment> segmentSupplier;
    private final String seqName;
    private final AtomicBoolean preloadTriggered = new AtomicBoolean(false);
    private final AtomicInteger staleResetCount = new AtomicInteger(0);
    private volatile Runnable asyncTaskSubmitter;

    public DoubleBuffer(String seqName, Supplier<Segment> segmentSupplier) {
        this.seqName = seqName;
        this.segmentSupplier = segmentSupplier;
        this.current = EmptySegment.INSTANCE;
        this.standby = EmptySegment.INSTANCE;
    }

    public void setAsyncTaskSubmitter(Runnable asyncTaskSubmitter) { this.asyncTaskSubmitter = asyncTaskSubmitter; }

    public long next() {
        if (current instanceof EmptySegment) {
            switchLock.lock();
            try { if (current instanceof EmptySegment) { current = loadSegment(); } }
            finally { switchLock.unlock(); }
        }
        int retryCount = 0;
        while (retryCount < MAX_RETRY) {
            long value = current.next();
            if (value >= 0) { checkPreload(); return value; }
            value = switchBuffer();
            if (value >= 0) { return value; }
            retryCount++;
        }
        throw new SequenceException(SequenceErrorCode.SEGMENT_LOAD_FAILED, ...);
    }

    private boolean needsPreload() { /* 双重条件：usage>=90% || remaining<max(50, totalSize*0.3) */ }
    private void checkPreload() { /* CAS preloadTriggered + stale 重置 + triggerAsyncLoad */ }
    private void triggerAsyncLoad() { /* 优先 asyncTaskSubmitter，降级同步 */ }
    public void loadStandby() { /* switchLock 保护下 segmentSupplier.get() */ }
    private long switchBuffer() { /* ReentrantLock + 双重检查 + 切换/同步加载 */ }
    private Segment loadSegment() { /* segmentSupplier.get() */ }

    public Segment getCurrent() { return current; }
    public Segment getStandby() { return standby; }
    public long remaining() { return current.remaining() + standby.remaining(); }
    public int getStaleResetCount() { return staleResetCount.get(); }
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| current | volatile Segment | 当前使用的 Segment，初始为 EmptySegment.INSTANCE |
| standby | volatile Segment | 备用 Segment，初始为 EmptySegment.INSTANCE |
| switchLock | ReentrantLock | 切换锁，保证切换原子性 |
| segmentSupplier | Supplier\<Segment\> | 号段加载函数（从数据库加载新号段） |
| seqName | String | 序列名称（用于日志） |
| preloadTriggered | AtomicBoolean | 预加载标记，CAS 保证仅触发一次 |
| staleResetCount | AtomicInteger | 活循环监控计数 |
| asyncTaskSubmitter | volatile Runnable | 异步任务提交器 |
| PRELOAD_THRESHOLD | static final double | 预加载触发阈值，固定 0.1（10%） |
| MAX_RETRY | static final int | next() 最大重试次数，固定 3 |

**方法说明**：

| 方法 | 说明 |
|------|------|
| next() | 核心分配方法。延迟初始化 → 从 current 分配 → 耗尽时 switchBuffer → 最多重试 3 次 |
| needsPreload() | 双重条件判断：使用率 >= 90% **或** 剩余 < max(50, 总大小 x 0.3) |
| checkPreload() | 检查并触发异步预加载。CAS 确保仅触发一次，支持 stale 标记重置（>10 次预警） |

#### 4.3.3 StrictSequence

**类路径**：`com.ccb.jx.seq.core.StrictSequence`

**职责**：实现严格连续模式。每次 `nextVal` 在 `REQUIRES_NEW` 事务中完成 `SELECT ... FOR UPDATE` → 计算下一个值 → `UPDATE` → `COMMIT`。完整支持 Oracle 语义：increment_by、min_value、max_value、cycle、start_with。实现 `SequenceStrategy` 接口。

**设计要点**：
- 使用 `TransactionTemplate` 编程式事务（非 `@Transactional`），解决自调用 AOP 代理失效问题
- 事务超时 3 秒
- `nextVal(seqName, config)` 接收序列名称和配置，在事务内通过 FOR UPDATE 重新读取配置
- 包含 `start_with` 修正逻辑：当 `max_id` 为 DDL 默认值（1）且 `start_with > 1` 时，修正 `currentMaxId`
- 使用 CAS `updateMaxId(seqName, newMaxId, originalMaxId)` 作为安全网
- FOR UPDATE 已保证串行化，无需 CAS 重试

```java
package com.ccb.jx.seq.core;

import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import com.ccb.jx.seq.model.SequenceConfig;
import com.ccb.jx.seq.repository.SequenceRepository;
import org.springframework.transaction.PlatformTransactionManager;
import org.springframework.transaction.TransactionDefinition;
import org.springframework.transaction.support.TransactionTemplate;

public class StrictSequence implements SequenceStrategy {

    private final SequenceRepository sequenceRepository;
    private final TransactionTemplate transactionTemplate;

    public StrictSequence(SequenceRepository sequenceRepository,
                           PlatformTransactionManager transactionManager) {
        this(sequenceRepository, transactionManager, 3);
    }

    public StrictSequence(SequenceRepository sequenceRepository,
                           PlatformTransactionManager transactionManager,
                           int transactionTimeoutSeconds) {
        this.sequenceRepository = sequenceRepository;
        this.transactionTemplate = new TransactionTemplate(transactionManager);
        this.transactionTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);
        this.transactionTemplate.setTimeout(transactionTimeoutSeconds);
    }

    @Override
    public long nextVal(String seqName, SequenceConfig config) {
        return nextValInternal(seqName, config);
    }

    private long nextValInternal(String seqName, SequenceConfig config) {
        return transactionTemplate.execute(status -> {
            SequenceConfig lockedConfig = sequenceRepository.selectConfigForUpdate(seqName);
            if (lockedConfig == null) { throw new SequenceException(SEQ_NOT_FOUND, ...); }

            long currentMaxId = lockedConfig.getMaxId();
            long originalMaxId = currentMaxId;
            // ... 读取配置、start_with 修正、计算下一个值、边界检查、CYCLE 回绕

            int updated = sequenceRepository.updateMaxId(seqName, newMaxId, originalMaxId);
            if (updated == 0) { throw new SequenceException(DB_ERROR, ...); }
            return nextVal;
        });
    }
}
```

**方法说明**：

| 方法 | 可见性 | 参数 | 返回值 | 事务 | 说明 |
|------|--------|------|--------|------|------|
| nextVal | public | String seqName, SequenceConfig config | long | TransactionTemplate REQUIRES_NEW, timeout=3s | 核心方法。FOR UPDATE 锁定 → 计算 → UPDATE → COMMIT |

**Oracle 语义映射**：

| Oracle 特性 | 实现方式 | 对应字段/逻辑 |
|-------------|----------|---------------|
| NEXTVAL | nextVal(seqName, config) | 统一入口 |
| INCREMENT BY n | 计算时 +incrementBy | sequence_config.increment_by |
| START WITH n | max_id=1 且 startWith>1 时修正 | sequence_config.start_with |
| MAXVALUE n | 越界检查 | sequence_config.max_value |
| MINVALUE n | 循环回绕目标值 | sequence_config.min_value |
| CYCLE | 越界且 cycle=1 时回绕 | sequence_config.cycle |
| NOCYCLE | 越界且 cycle=0 时抛异常 | sequence_config.cycle=0 |
| CURRVAL | SessionHolder.getCurrVal() | session 层 ThreadLocal |

#### 4.3.4 CachedSequence

#### 4.3.5 NativeStrictSequence

**类路径**：`com.ccb.jx.seq.core.NativeStrictSequence`

**职责**：委托数据库原生 Sequence 对象（如 TDSQL 的 `tdsql_nextval()`）生成连续值，消除应用层 `SELECT ... FOR UPDATE` 行锁竞争。实现 `SequenceStrategy` 接口。

**设计要点**：
- Level-2 懒创建自愈：当 `dialect.nextVal()` 抛出"Sequence 不存在"异常时，自动调用 `dialect.createSequence()` 并重试
- best-effort max_id 更新：更新成功不影响正确性，失败仅打 WARN 日志，不阻塞返回值
- SEQ_EXHAUSTED 检查：在调用 `dialect.nextVal()` 前检查 maxValue 边界
- 不使用 TransactionTemplate（依赖 DB 原生序列的原子性）

```java
package com.ccb.jx.seq.core;

import com.ccb.jx.seq.dialect.NativeSequenceDialect;
import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import com.ccb.jx.seq.model.SequenceConfig;
import com.ccb.jx.seq.repository.SequenceRepository;

public class NativeStrictSequence implements SequenceStrategy {

    private final NativeSequenceDialect dialect;
    private final SequenceRepository repository;

    public NativeStrictSequence(NativeSequenceDialect dialect, SequenceRepository repository) {
        this.dialect = dialect;
        this.repository = repository;
    }

    @Override
    public long nextVal(String seqName, SequenceConfig config) {
        // 1. 检查序列是否已耗尽
        if (config.getMaxId() >= config.getMaxValue() && config.getCycle() != 1) {
            throw new SequenceException(SequenceErrorCode.SEQ_EXHAUSTED, ...);
        }

        // 2. 委托数据库原生 Sequence 生成值
        long value;
        try {
            value = dialect.nextVal(config);
        } catch (SequenceException e) {
            // Level 2 懒创建自愈
            if (dialect.isSequenceNotExist(e)) {
                dialect.createSequence(config);
                value = dialect.nextVal(config);
            } else {
                throw e;
            }
        }

        // 3. best-effort 更新 max_id
        try {
            repository.updateMaxId(seqName, value, config.getMaxId());
        } catch (Exception e) {
            log.warn("[NATIVE] Failed to update max_id: ...", e.getMessage());
        }

        return value;
    }
}
```

**方法说明**：

| 方法 | 可见性 | 说明 |
|------|--------|------|
| nextVal(String, SequenceConfig) | public | 实现 SequenceStrategy。耗尽检查 → dialect.nextVal → 懒创建自愈 → best-effort 更新 max_id |

#### 4.3.6 SequenceService

**职责**：对外统一 API 门面。根据序列配置中的 mode 字段通过策略映射路由到对应的 SequenceStrategy 实现。维护配置缓存（ConcurrentHashMap）和定时刷新调度器。实现 `nextVal()` 和 `currVal()`。

**设计要点**：

```java
import com.ccb.jx.seq.session.SessionHolder;
import java.util.Collections;
import java.util.List;
import java.util.Map;
import java.util.Objects;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.ConcurrentMap;
import java.util.concurrent.Executors;
import java.util.concurrent.ScheduledExecutorService;
import java.util.concurrent.TimeUnit;

public class SequenceService {

    private final Map<Mode, SequenceStrategy> strategyMap;
    private final SequenceRepository sequenceRepository;
    private final ConcurrentMap<String, SequenceConfig> configCache = new ConcurrentHashMap<>();
    private final ScheduledExecutorService cacheRefreshScheduler;
    private final long configCacheTtlMinutes;

    public SequenceService(Map<Mode, SequenceStrategy> strategyMap,
                           SequenceRepository sequenceRepository) {
        this(strategyMap, sequenceRepository, 5L);
    }

    public SequenceService(Map<Mode, SequenceStrategy> strategyMap,
                           SequenceRepository sequenceRepository,
                           long configCacheTtlMinutes) {
        this.strategyMap = Collections.unmodifiableMap(Objects.requireNonNull(strategyMap));
        this.sequenceRepository = Objects.requireNonNull(sequenceRepository);
        if (configCacheTtlMinutes <= 0) { throw new IllegalArgumentException(...); }
        this.configCacheTtlMinutes = configCacheTtlMinutes;
        this.cacheRefreshScheduler = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "sequence-config-cache-refresh");
            t.setDaemon(true);
            return t;
        });
        ConfigCacheRefresher refresher = new ConfigCacheRefresher(
                getCachedSequence(), sequenceRepository, configCache);
        cacheRefreshScheduler.scheduleAtFixedRate(
                refresher, configCacheTtlMinutes, configCacheTtlMinutes, TimeUnit.MINUTES);
    }

    public long nextVal(String seqName) {
        Objects.requireNonNull(seqName, "seqName must not be null");
        SequenceConfig config = getConfig(seqName);
        SequenceStrategy strategy = strategyMap.get(config.getMode());
        if (strategy == null) { throw new SequenceException(INVALID_CONFIG, ...); }
        long value = strategy.nextVal(seqName, config);
        SessionHolder.setCurrVal(seqName, value);
        return value;
    }

    public long currVal(String seqName) { return SessionHolder.getCurrVal(seqName); }

    public SequenceConfig getConfig(String seqName) {
        return configCache.computeIfAbsent(seqName, name -> {
            SequenceConfig config = sequenceRepository.selectConfig(name);
            if (config == null) {
                // Level 1 懒创建：INSERT IGNORE
                sequenceRepository.insertConfigIgnore(createDefaultConfig(name));
                config = sequenceRepository.selectConfig(name);
                if (config == null) { throw new SequenceException(SEQ_NOT_FOUND, ...); }
            }
            return config;
        });
    }

    public List<SequenceConfig> getAllConfigs() { return sequenceRepository.selectAllConfig(); }
    public void evictConfig(String seqName) { configCache.remove(seqName); }
    public void clearCache() { configCache.clear(); }
    public void stop() { /* 关闭 cacheRefreshScheduler，3s 超时 */ }
    public CachedSequence getCachedSequence() {
        SequenceStrategy strategy = strategyMap.get(Mode.CACHED);
        return strategy instanceof CachedSequence ? (CachedSequence) strategy : null;
    }
    public SequenceInfo getSequenceInfo(String seqName) { /* 综合快照 */ }

    public static class SequenceInfo {
        private final SequenceConfig config;
        private final long currVal;
        private final boolean hasCurrVal;
        private final DoubleBuffer buffer;
        // constructor + getters + getRemaining()
    }
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|

| sequenceRepository | SequenceRepository | 数据访问层（用于查询配置和清理旧记录） |
| configCache | ConcurrentMap\<String, SequenceConfig\> | 配置缓存，computeIfAbsent 安全懒加载 |

**方法说明**：

| 方法 | 返回值 | 说明 |
|------|--------|------|
| nextVal(String) | long | 统一入口。校验 → 取配置 → 策略路由 → 更新会话 → 返回 |
| currVal(String) | long | 会话级当前值。委托 SessionHolder.getCurrVal() 静态方法 |

**模式路由逻辑**：

```
nextVal(seqName)
  │
  ├── seqName 非空校验（Objects.requireNonNull）
  │
  ├── getConfig(seqName)
  │    ├── 缓存命中 → 返回
  │    └── 缓存未命中 → computeIfAbsent
  │          ├── selectConfig(seqName) from DB
  │          ├── 不存在 → INSERT IGNORE 默认配置 → 重新查询
  │          └── 写入缓存，返回
  │
  ├── strategyMap.get(config.mode)
  │
  ├── mode == STRICT + dialect=NONE
  │    └── StrictSequence.nextVal(seqName, config)
  │         └── TransactionTemplate(REQUIRES_NEW, timeout=3s)
  │              ├── SELECT ... FOR UPDATE
  │              ├── start_with 修正（max_id==1 && startWith>1）
  │              ├── 计算下一个值（边界检查、CYCLE 回绕）
  │              ├── CAS updateMaxId(seqName, newMaxId, originalMaxId)
  │              └── COMMIT
  │
  ├── mode == STRICT + dialect=TDSQL
  │    └── NativeStrictSequence.nextVal(seqName, config)
  │         ├── 耗尽检查（maxId >= maxValue && !cycle）
  │         ├── dialect.nextVal(config)
  │         │    └── SELECT tdsql_nextval(?)
  │         ├── 懒创建自愈（isSequenceNotExist → createSequence → retry）
  │         └── best-effort updateMaxId
  │
  ├── mode == CACHED
  │    └── CachedSequence.nextVal(seqName)
  │         ├── checkShuttingDown() → 抛出 SHUTTING_DOWN
  │         ├── initBuffer(seqName)  ← computeIfAbsent 安全初始化
  │         ├── buffer.next()  ← 内存分配（AtomicLong.getAndIncrement）
  │         ├── checkPreload() → 异步加载备用号段
  │         └── 返回
  │
  └── SessionHolder.setCurrVal(seqName, value)
       └── 返回 value
```

#### 4.3.7 ConfigCacheRefresher

**类路径**：`com.ccb.jx.seq.core.ConfigCacheRefresher`

**职责**：配置缓存定时刷新任务，由 `SequenceService` 的定时调度器周期性调用。遵循单一职责原则，将刷新逻辑从 SequenceService 中抽离。

**设计要点**：
- 实现 `Runnable`，可直接传给 `ScheduledExecutorService.scheduleAtFixedRate`
- 3 步刷新：cleanupStaleBuffers → refreshConfigCache → cleanupOldRecords
- `refreshFailCount`（AtomicInteger）记录连续失败次数，>=3 时升级为 ERROR 日志
- `cachedSequence` 可能为 null（NATIVE 模式下 strategyMap 中无 CachedSequence）

```java
package com.ccb.jx.seq.core;

import com.ccb.jx.seq.model.SequenceConfig;
import com.ccb.jx.seq.repository.SequenceRepository;
import java.util.Calendar;
import java.util.Date;
import java.util.HashSet;
import java.util.List;
import java.util.Set;
import java.util.concurrent.ConcurrentMap;
import java.util.concurrent.atomic.AtomicInteger;

public class ConfigCacheRefresher implements Runnable {

    private final CachedSequence cachedSequence;
    private final SequenceRepository sequenceRepository;
    private final ConcurrentMap<String, SequenceConfig> configCache;
    private final AtomicInteger refreshFailCount = new AtomicInteger(0);

    public ConfigCacheRefresher(CachedSequence cachedSequence,
                                SequenceRepository sequenceRepository,
                                ConcurrentMap<String, SequenceConfig> configCache) {
        this.cachedSequence = cachedSequence;
        this.sequenceRepository = sequenceRepository;
        this.configCache = configCache;
    }

    @Override
    public void run() {
        try {
            cleanupStaleBuffers();
            refreshConfigCache();
            cleanupOldRecords();
            refreshFailCount.set(0);
        } catch (Exception e) {
            int failures = refreshFailCount.incrementAndGet();
            if (failures >= 3) { log.error(...); } else { log.warn(...); }
        }
    }

    void cleanupStaleBuffers() { /* selectAllConfig → evictStaleBuffers */ }
    void refreshConfigCache() { configCache.clear(); }
    void cleanupOldRecords() { /* 删除 30 天前的 sequence_record */ }
    int getRefreshFailCount() { return refreshFailCount.get(); }
}
```

**3 步刷新流程**：

| 步骤 | 方法 | 说明 |
|------|------|------|
| 1 | cleanupStaleBuffers() | 查询全量配置，清理已从配置表中删除的序列的 DoubleBuffer |
| 2 | refreshConfigCache() | 粗粒度清空整个缓存，下次访问时从 DB 重新加载 |
| 3 | cleanupOldRecords() | 删除 30 天前的旧 sequence_record 记录 |

#### 4.4.2 TdsqlSequenceDialect

**类路径**：`com.ccb.jx.seq.dialect.TdsqlSequenceDialect`

**职责**：TDSQL 原生 Sequence 方言实现，使用 JdbcTemplate（非 MyBatis）执行 SQL。

**设计要点**：
- 使用 `JdbcTemplate`（非 MyBatis），因为原生 Sequence 操作是 DDL/特殊函数调用，不适合 MyBatis 映射
- SQL 模板：
  - 获取值：`SELECT tdsql_nextval(?)`
  - 创建：`CREATE TDSQL_SEQUENCE seqName START WITH n TDSQL_MINVALUE n [TDSQL_NOMAXVALUE|TDSQL_MAXVALUE n] TDSQL_INCREMENT BY n [TDSQL_CYCLE|TDSQL_NOCYCLE]`
  - 删除：`DROP TDSQL_SEQUENCE seqName`
  - 检查：`SELECT tdsql_lastval(?)`
- NOMAXVALUE 处理：`maxValue >= Long.MAX_VALUE - 1` 时使用 `TDSQL_NOMAXVALUE`
- `isSequenceNotExist` 遍历 cause 链检查：消息包含 "NOT EXIST"/"NOT EXISTS"/"1146"/"ER_NO_SUCH_TABLE"，SQLState 为 "42S02"/"42S22"

```java
package com.ccb.jx.seq.dialect;

import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import com.ccb.jx.seq.model.SequenceConfig;
import org.springframework.jdbc.core.JdbcTemplate;
import org.springframework.dao.DataAccessException;
import java.sql.SQLException;

public class TdsqlSequenceDialect implements NativeSequenceDialect {

    private final JdbcTemplate jdbcTemplate;

    public TdsqlSequenceDialect(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public long nextVal(SequenceConfig config) {
        try {
            Long value = jdbcTemplate.queryForObject("SELECT tdsql_nextval(?)", Long.class, config.getSeqName());
            if (value == null) { throw new SequenceException(NATIVE_SEQ_ERROR, ...); }
            return value;
        } catch (DataAccessException e) {
            throw new SequenceException(NATIVE_SEQ_ERROR, ..., e);
        }
    }

    @Override
    public void createSequence(SequenceConfig config) {
        StringBuilder ddl = new StringBuilder("CREATE TDSQL_SEQUENCE ");
        ddl.append(config.getSeqName());
        ddl.append(" START WITH ").append(config.getStartWith());
        ddl.append(" TDSQL_MINVALUE ").append(config.getMinValue());
        if (config.getMaxValue() >= Long.MAX_VALUE - 1) {
            ddl.append(" TDSQL_NOMAXVALUE");
        } else {
            ddl.append(" TDSQL_MAXVALUE ").append(config.getMaxValue());
        }
        ddl.append(" TDSQL_INCREMENT BY ").append(config.getIncrementBy());
        ddl.append(config.getCycle() == 1 ? " TDSQL_CYCLE" : " TDSQL_NOCYCLE");
        jdbcTemplate.execute(ddl.toString());
    }

    @Override
    public void dropSequence(String seqName) {
        jdbcTemplate.execute("DROP TDSQL_SEQUENCE " + seqName);
    }

    @Override
    public boolean sequenceExists(String seqName) {
        try {
            jdbcTemplate.queryForObject("SELECT tdsql_lastval(?)", Long.class, seqName);
            return true;
        } catch (DataAccessException e) { return false; }
    }

    @Override
    public boolean isSequenceNotExist(SequenceException e) {
        // 遍历 cause 链，检查消息和 SQLState
        Throwable cause = e.getCause();
        while (cause != null) {
            String msg = cause.getMessage();
            if (msg != null) {
                String upper = msg.toUpperCase();
                if (upper.contains("NOT EXIST") || upper.contains("NOT EXISTS")
                        || upper.contains("1146") || upper.contains("ER_NO_SUCH_TABLE")) {
                    return true;
                }
            }
            if (cause instanceof SQLException) {
                String sqlState = ((SQLException) cause).getSQLState();
                if ("42S02".equals(sqlState) || "42S22".equals(sqlState)) { return true; }
            }
            cause = cause.getCause();
        }
        return false;
    }
}
```

#### 4.4.3 NativeSequenceSynchronizer

**类路径**：`com.ccb.jx.seq.dialect.NativeSequenceSynchronizer`

**职责**：在应用启动时，扫描 `sequence_config` 表中所有 `mode=STRICT` 的序列，确保对应的数据库原生 Sequence 对象已存在于 DB 中。

```java
package com.ccb.jx.seq.dialect;

import com.ccb.jx.seq.model.Mode;
import com.ccb.jx.seq.model.SequenceConfig;
import com.ccb.jx.seq.repository.SequenceRepository;
import java.util.List;

public class NativeSequenceSynchronizer {

    private final SequenceRepository repository;
    private final NativeSequenceDialect dialect;

    public NativeSequenceSynchronizer(SequenceRepository repository, NativeSequenceDialect dialect) {
        this.repository = repository;
        this.dialect = dialect;
    }

    public void syncOnStartup() {
        List<SequenceConfig> allConfigs = repository.selectAllConfig();
        int created = 0, skipped = 0;
        for (SequenceConfig config : allConfigs) {
            if (config.getMode() != Mode.STRICT) { continue; }
            if (dialect.sequenceExists(config.getSeqName())) { skipped++; }
            else { dialect.createSequence(config); created++; }
        }
        log.info("[NATIVE-SYNC] Startup sync complete: total={}, created={}, skipped={}",
                allConfigs.size(), created, skipped);
    }
}
```

**说明**：`syncOnStartup()` 在 `SequenceAutoConfiguration.nativeStrictSequence()` Bean 创建时直接调用，无需等待初始化回调。

### 4.5 session 包：会话语义

#### 4.5.1 SessionHolder

**类路径**：`com.ccb.jx.seq.session.SessionHolder`

**职责**：通过 ThreadLocal 实现 Oracle CURRVAL 会话语义。每个 HTTP 请求拥有独立的 currVal 映射。

**设计要点**：
- `final` 工具类，私有构造器（`throw new AssertionError()`，More Effective Java 第 4 条）
- 所有方法均为 `static`，无需实例化
- `CURRVAL_HOLDER` 使用 `ThreadLocal.withInitial(HashMap::new)` 懒初始化
- `getCurrVal(seqName)` 未初始化时抛出 `SequenceException(CURRVAL_NOT_INITIALIZED)`
- 提供 `wrap(Runnable)` 和 `wrap(Callable)` 方法，用于非 HTTP 线程场景的自动清理

```java
package com.ccb.jx.seq.session;

import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.Callable;

public final class SessionHolder {

    private static final ThreadLocal<Map<String, Long>> CURRVAL_HOLDER =
            ThreadLocal.withInitial(HashMap::new);

    private SessionHolder() { throw new AssertionError("工具类不允许实例化"); }

    public static void setCurrVal(String seqName, long value) { CURRVAL_HOLDER.get().put(seqName, value); }

    public static long getCurrVal(String seqName) {
        Map<String, Long> map = CURRVAL_HOLDER.get();
        Long value = map.get(seqName);
        if (value == null) { throw new SequenceException(SequenceErrorCode.CURRVAL_NOT_INITIALIZED, "seqName=" + seqName); }
        return value;
    }

    public static void clear() { CURRVAL_HOLDER.remove(); }

    public static Runnable wrap(Runnable task) {
        return () -> { try { task.run(); } finally { clear(); } };
    }

    public static <T> Callable<T> wrap(Callable<T> task) {
        return () -> { try { return task.call(); } finally { clear(); } };
    }
}
```

```java
import com.ccb.jx.seq.exception.SequenceErrorCode;
import com.ccb.jx.seq.exception.SequenceException;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;
import java.util.LinkedHashMap;
import java.util.Map;

@RestControllerAdvice
public class SequenceExceptionHandler {

    @ExceptionHandler(SequenceException.class)
    public ResponseEntity<Map<String, Object>> handleSequenceException(SequenceException e) {
        HttpStatus status = resolveHttpStatus(e.getErrorCode());
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("code", e.getErrorCode().getCode());
        body.put("message", e.getMessage());
        return ResponseEntity.status(status).body(body);
    }

    private HttpStatus resolveHttpStatus(SequenceErrorCode errorCode) {
        switch (errorCode) {
            case SEQ_NOT_FOUND: return HttpStatus.NOT_FOUND;
            case CURRVAL_NOT_INITIALIZED: return HttpStatus.BAD_REQUEST;
            case SEQ_EXHAUSTED: return HttpStatus.GONE;
            case SHUTTING_DOWN: return HttpStatus.SERVICE_UNAVAILABLE;
            default: return HttpStatus.INTERNAL_SERVER_ERROR;
        }
    }
}
```

#### 4.9.3 SequenceShutdownHook

**类路径**：`com.ccb.jx.seq.config.SequenceShutdownHook`

**职责**：监听 Spring 容器关闭事件（`ContextClosedEvent` + `@PreDestroy`），执行优雅关闭流程。

**设计要点**：
- 实现 `ApplicationListener<ContextClosedEvent>` + `@PreDestroy` 双入口
- `AtomicBoolean shutdownCompleted` — 保证关闭逻辑无论触发几次都只执行一次
- 4 步关闭：transitionTo(SHUTTING_DOWN) → shutdown executor (5s timeout) → persistUnusedSegments → transitionTo(SHUTDOWN)

```java
package com.ccb.jx.seq.config;

import com.ccb.jx.seq.core.CachedSequence;
import com.ccb.jx.seq.core.ServiceState;
import org.springframework.context.ApplicationListener;
import org.springframework.context.event.ContextClosedEvent;
import javax.annotation.PreDestroy;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.AtomicBoolean;

public class SequenceShutdownHook implements ApplicationListener<ContextClosedEvent> {

    private static final int SHUTDOWN_TIMEOUT_SECONDS = 5;
    private final CachedSequence cachedSequence;
    private final AtomicBoolean shutdownCompleted = new AtomicBoolean(false);

    public SequenceShutdownHook(CachedSequence cachedSequence) { this.cachedSequence = cachedSequence; }

    @Override public void onApplicationEvent(ContextClosedEvent event) { doShutdown(); }
    @PreDestroy public void destroy() { doShutdown(); }

    private void doShutdown() {
        if (!shutdownCompleted.compareAndSet(false, true)) { return; }
        // Step 1: 状态转换到 SHUTTING_DOWN
        cachedSequence.transitionTo(ServiceState.SHUTTING_DOWN);
        // Step 2: 关闭异步预加载线程池（5s 超时）
        ExecutorService executor = cachedSequence.getAsyncExecutor();
        executor.shutdown();
        try {
            if (!executor.awaitTermination(SHUTDOWN_TIMEOUT_SECONDS, TimeUnit.SECONDS)) { executor.shutdownNow(); }
        } catch (InterruptedException e) { Thread.currentThread().interrupt(); executor.shutdownNow(); }
        // Step 3: 持久化未使用号段信息
        cachedSequence.persistUnusedSegments();
        // Step 4: 状态转换到 SHUTDOWN
        cachedSequence.transitionTo(ServiceState.SHUTDOWN);
    }
}
```

### 6.4 数据状态流转

**sequence_record.status 状态机**：

```

     ▼                     ▼
```

|------|------|----------|----------|

| EXPIRED | 已过期 | 号段耗尽或过期标记 | 历史记录保留 |

## 7. 算法设计

### 7.1 动态步长调整算法

**算法来源**：美团 Leaf 线性调整算法 + 阻尼机制。

**目标**：使数据库号段更新周期稳定在 15min 左右。

**公式**：

```
ratio = SEGMENT_DURATION_MINUTES / actualDurationMinutes
targetStep = round(currentStep * ratio)

// 阻尼：单次调整不超过 2 倍
maxStep = currentStep * 2
minStep = ceil(currentStep / 2.0)
targetStep = max(minStep, min(maxStep, targetStep))

// 全局范围约束
targetStep = max(MIN_STEP, min(MAX_STEP, targetStep))

SEGMENT_DURATION_MINUTES = 15
MIN_STEP = 100
MAX_STEP = 1_000_000
```

**调整策略**：

| actualDuration | 含义 | nextStep 变化 |
|---------------|------|---------------|
| 5min | 消耗过快 | step x 2（阻尼上限） |
| 15min | 消耗适中 | step x 1（步长不变） |
| 30min | 消耗较慢 | step x 0.5（阻尼下限） |
| 60min | 消耗很慢 | step x 0.5（阻尼下限） |

**收敛特性**：经过 3-5 个周期后步长自动收敛，使更新间隔稳定在 15min 左右。阻尼机制防止步长振荡。

**实现要点**：
- 在每次加载新号段时计算（`loadSegment` 方法内）
- 使用浮点数计算保留精度，避免 `toMinutes` 截断导致步长调整不准
- `actualDurationMinutes` 最小值为 1.0，防止除零或极端放大
- 步长更新仅在内存中进行，不持久化到 DB

### 7.2 双 Buffer 切换算法

**核心思想**：两个 Segment 交替使用，零等待切换。

```
状态定义:
  current: 当前使用的 Segment（volatile）
  standby: 备用 Segment（volatile）
  preloadTriggered: 预加载是否已触发（AtomicBoolean）

分配路径（next）:
  1. current instanceof EmptySegment → switchLock 保护下同步加载
  2. current.next()  ← AtomicLong.getAndIncrement() 无锁
  3. 如果返回 >= 0:
     a. checkPreload(): needsPreload()?
     b. needsPreload() = usage() >= 90% || remaining < max(50, totalSize x 0.3)
     c. 是则 CAS preloadTriggered false→true 并提交异步加载任务
     d. stale 检测：preloadTriggered=true 但 standby=EmptySegment → 重置标记
     e. 返回 value
  4. 如果返回 -1（耗尽）:
     a. 进入 switchBuffer()
     b. 获取 ReentrantLock
     c. 双重检查 current 是否确实耗尽
     d. standby.isInitialized():
        - current = standby, standby = EmptySegment.INSTANCE
        - preloadTriggered = false, staleResetCount = 0
        - 从新 current 分配并返回
     e. standby 未初始化: 同步加载新 Segment
        - current = newSegment, preloadTriggered = false
        - 从新 current 分配并返回
  5. 释放 ReentrantLock
  6. 最多重试 MAX_RETRY=3 次

预加载路径（异步线程）:
  1. asyncTaskSubmitter.run() → 提交 loadStandby() 到线程池
  2. loadStandby() 在 switchLock 保护下:
     a. standby 非 EmptySegment → 跳过
     b. standby 为 EmptySegment → segmentSupplier.get() → standby = segment
  3. segmentSupplier 由 CachedSequence 提供:
     a. TransactionTemplate(REQUIRES_NEW) 执行 loadSegment
     b. SELECT ... FOR UPDATE → CAS updateMaxId → INSERT record
     c. return new Segment(startValue, endValue)
```

**性能模型**：

| 路径 | 触发条件 | 耗时 | 占比 |
|------|----------|------|------|
| 正常分配 | 当前 Segment 未耗尽 | < 1us | 99.9% |
| 切换分配 | 当前耗尽，备用可用 | < 100us | < 0.1% |
| 同步加载 | 备用未就绪（兜底） | ~10ms | 极低 |

**预加载覆盖保障**：
- 触发阈值：使用率 >= 90% **或** 剩余 < max(50, 总大小 x 0.3)
- 异步加载耗时：~10ms
- 剩余号消耗时间：100 / (5000/1000ms) = 20ms（5000 TPS）
- 预加载远快于消耗速度，100% 覆盖

### 7.3 号段加载算法

**算法位于**：`CachedSequence.loadSegment()`

```
loadSegment(seqName):
  1. TransactionTemplate.execute(REQUIRES_NEW, timeout=3s)
  
  2. SELECT ... FOR UPDATE
     FROM sequence_config WHERE seq_name = ?
     -- 行级悲观锁，保证号段加载原子性
  
  3. 计算动态步长（非首次加载时）
     actualDuration = System.currentTimeMillis() - currentSegment.getLoadTime()
     step = calculateDynamicStep(config.step, actualDuration)
  
  4. 基于 FOR UPDATE 锁保护，读取当前 max_id 并计算新值
     oldMaxId = config.maxId
     newMaxId = oldMaxId + step
     
     CYCLE 和 maxValue 边界检查:
     IF newMaxId > maxValue:
       IF cycle: newMaxId = minValue + step - 1
       ELSE: throw SEQ_EXHAUSTED
  
  5. CAS 更新: UPDATE sequence_config SET max_id = #{newMaxId}
     WHERE seq_name = #{seqName} AND max_id = #{oldMaxId}
     -- 影响行数为 0 → 抛出 DB_ERROR
  
  6. startValue = newMaxId - step + 1
  
  7. INSERT INTO sequence_record
     (seq_name, start_value, end_value, instance_id, status, alloc_time)
     VALUES (?, ?, ?, ?, 'ALLOCATED', NOW())
     -- 审计追踪
  
  8. COMMIT
  
  9. return new Segment(startValue, newMaxId)
```

**核心保证**：
- FOR UPDATE 和 CAS `updateMaxId` 双重保护，保证号段分配原子性
- UPDATE 和 INSERT 在同一事务，保证数据一致性
- UPDATE 影响行数为 0 时抛出 `DB_ERROR` 异常，事务回滚
- 号段范围 = `[newMaxId - step + 1, newMaxId]`，确保不重复

### 7.4 严格连续模式计算算法

**算法位于**：`StrictSequence.nextVal()`

```
nextVal(seqName, config):
  TransactionTemplate.execute(REQUIRES_NEW, timeout=3s):
  
  1. SELECT ... FOR UPDATE WHERE seq_name = ?
     -- 不存在 → 抛出 SEQ_NOT_FOUND
  
  2. 读取配置
     currentMaxId = lockedConfig.maxId
     originalMaxId = currentMaxId
     incrementBy = lockedConfig.incrementBy
     minValue = lockedConfig.minValue
     maxValue = lockedConfig.maxValue
     cycle = (lockedConfig.cycle == 1)
     startWith = lockedConfig.startWith
  
  3. start_with 修正
     IF currentMaxId == 1 AND startWith > 1:
       currentMaxId = startWith - incrementBy
       -- 修正后 nextVal = (startWith - incrementBy) + incrementBy = startWith
  
  4. 计算下一个值
     candidate = currentMaxId + incrementBy
     
     IF candidate > maxValue:
       IF cycle:
         nextVal = minValue
         newMaxId = minValue
       ELSE:
         throw SEQ_EXHAUSTED
     ELSE:
       nextVal = candidate
       newMaxId = nextVal
  
  5. CAS 更新: UPDATE sequence_config SET max_id = #{newMaxId}
     WHERE seq_name = #{seqName} AND max_id = #{originalMaxId}
     -- 影响行数为 0 → 抛出 DB_ERROR
  
  6. COMMIT → 释放行锁
  
  7. return nextVal
```

**边界条件**：

| 场景 | currentMaxId | incrementBy | maxValue | cycle | 结果 |
|------|-------------|-------------|----------|-------|------|
| 正常递增 | 100 | 1 | Long.MAX_VALUE | false | 101 |
| 自定义步长 | 100 | 5 | Long.MAX_VALUE | false | 105 |
| start_with 修正 | 1 (DDL默认) | 1 | Long.MAX_VALUE | false | startWith |
| 到达最大值（循环） | 999999 | 1 | 1000000 | true | minValue |
| 到达最大值（非循环） | 1000000 | 1 | 1000000 | false | 抛出异常 |

### 7.5 NATIVE 模式 Level-2 懒创建自愈算法

**算法位于**：`NativeStrictSequence.nextVal()`

```
nextVal(seqName, config):
  1. 耗尽检查
     IF config.maxId >= config.maxValue AND config.cycle != 1:
       throw SEQ_EXHAUSTED

  2. 委托原生 Sequence 生成值
     TRY:
       value = dialect.nextVal(config)
       -- SELECT tdsql_nextval(?)
     CATCH SequenceException e:
       IF dialect.isSequenceNotExist(e):
         -- Level 2 懒创建自愈
         dialect.createSequence(config)
         -- CREATE TDSQL_SEQUENCE ...
         value = dialect.nextVal(config)
         -- 重试成功
       ELSE:
         throw e

  3. best-effort 更新 max_id
     TRY:
       repository.updateMaxId(seqName, value, config.maxId)
     CATCH Exception e:
       log.warn("[NATIVE] Failed to update max_id: ...")
       -- 失败不影响返回值

  4. return value
```

**两层懒创建**：

| 层级 | 位置 | 触发条件 | 操作 |
|------|------|----------|------|
| Level 1 | SequenceService.getConfig() | sequence_config 行不存在 | INSERT IGNORE 默认配置行 |
| Level 2 | NativeStrictSequence.nextVal() | DB Sequence 对象不存在 | createSequence + retry nextVal |

**isSequenceNotExist 判断逻辑**：
- 遍历异常 cause 链
- 消息包含 "NOT EXIST" / "NOT EXISTS" / "1146" / "ER_NO_SUCH_TABLE"
- SQLState 为 "42S02" / "42S22"

## 10. 异常处理设计

### 10.1 异常层次结构

```
RuntimeException
├── NullPointerException            # seqName 为 null
└── SequenceException              # 序列服务统一异常
      ├── errorCode: SequenceErrorCode
      │     ├── SEQ_NOT_FOUND (SEQ_001)
      │     ├── SEQ_EXHAUSTED (SEQ_002)
      │     ├── CURRVAL_NOT_INITIALIZED (SEQ_003)
      │     ├── SHUTTING_DOWN (SEQ_004)
      │     ├── DB_ERROR (SEQ_005)
      │     ├── INVALID_CONFIG (SEQ_006)
      │     ├── SEGMENT_LOAD_FAILED (SEQ_007)
      │     └── NATIVE_SEQ_ERROR (SEQ_008)
      └── (可选) cause: Throwable    # 根因异常
```

### 10.2 异常场景与处理策略

| 序号 | 异常场景 | 抛出位置 | 错误码 | 处理策略 |
|------|----------|----------|--------|----------|
| 1 | seqName 为 null | SequenceService.nextVal | -（NullPointerException） | 快速失败，调用方需修复代码 |
| 2 | 序列配置在 DB 中不存在 | SequenceService.getConfig | SEQ_NOT_FOUND | Level 1 懒创建（INSERT IGNORE），创建失败则快速失败 |
| 3 | FOR UPDATE 查询序列不存在 | StrictSequence.nextVal | SEQ_NOT_FOUND | 事务回滚，快速失败 |
| 4 | 非循环模式达到 max_value | StrictSequence / CachedSequence / NativeStrictSequence | SEQ_EXHAUSTED | 事务回滚，调用方需评估是否需重建序列 |
| 5 | CAS updateMaxId 影响行数为 0 | StrictSequence.nextVal / CachedSequence.loadSegment | DB_ERROR | 事务回滚，记录 ERROR 日志 |
| 6 | 服务关闭期间收到 nextVal 请求 | CachedSequence.nextVal | SHUTTING_DOWN | 快速失败，调用方需重试 |
| 7 | 未调用 nextVal 直接调用 currVal | SessionHolder.getCurrVal | CURRVAL_NOT_INITIALIZED | 快速失败，调用方需先调用 nextVal |
| 8 | 异步预加载异常 | DoubleBuffer.loadStandby | SEGMENT_LOAD_FAILED | 记录 ERROR 日志，备用 segment 可能不刷新 |
| 9 | 线程池任务被拒绝 | DoubleBuffer.checkPreload | -（stale 重置） | stale 标记重置后下次重试，>10 次预警 |
| 10 | 序列名称超长 | SequenceController.validateSeqName | INVALID_CONFIG | 快速失败，返回 400 |
| 11 | 原生序列操作失败 | TdsqlSequenceDialect | NATIVE_SEQ_ERROR | 快速失败，含根因异常 |
| 12 | 原生 Sequence 不存在 | NativeStrictSequence.nextVal | -（先捕获） | Level 2 懒创建自愈：createSequence + retry |
| 13 | best-effort max_id 更新失败 | NativeStrictSequence.nextVal | -（仅 WARN） | 记录 WARN 日志，不阻塞返回值 |

### 10.3 异常信息格式

```java

    errorCode=SEQ_NOT_FOUND(SEQ_001),
    message="序列不存在: seqName=ORDER_NO"

```

### 10.4 日志输出规范

| 异常级别 | 场景 | 输出内容 |
|----------|------|----------|
| ERROR | DB 操作失败、号段加载失败、原生序列操作失败 | `[STRICT/CACHED/NATIVE] 操作失败, seqName={}, msg={}` |

| DEBUG | 正常 nextVal、配置缓存加载、预加载触发 | `[SERVICE/STRICT/CACHED/BUFFER] 操作成功, seqName={}, detail={}` |

### 12.2 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| 严格连续 | Strictly Sequential | 多实例环境下序列号严格递增（1,2,3...） |
| 号段 | Segment | 从数据库批量预取的一段连续序列号范围 [start, end] |
| 双 Buffer | Double Buffer | 当前号段 + 备用号段交替使用，零等待切换 |
| 步长 | Step | 每次从 DB 预取的号段长度 |
| 动态步长 | Dynamic Step | 根据消耗速率自动调整步长的算法 |
| 阻尼 | Damping | 单次步长调整不超过 2 倍，防止步长振荡 |
| 会话语义 | Session Semantics | currVal 仅在当前会话（HTTP 请求）内有效 |
| 优雅关闭 | Graceful Shutdown | 应用关闭时完成当前请求、持久化号段再退出 |
| 持久化标记 | Persistence Marker | 号段分配时在 sequence_record 中插入的记录 |
| REQUIRES_NEW | REQUIRES_NEW | Spring 事务传播行为，挂起当前事务开启新事务 |
| TransactionTemplate | TransactionTemplate | Spring 编程式事务模板，解决自调用 @Transactional 失效问题 |
| 原生序列 | Native Sequence | 数据库原生 Sequence 对象（如 TDSQL 的 tdsql_nextval） |
| 懒创建自愈 | Lazy-Create Self-Healing | 运行时检测到 Sequence 不存在时自动创建并重试 |
| 空号段 | EmptySegment | Null Object 模式，表示尚未加载号段的状态 |
| 策略模式 | Strategy Pattern | 通过 Map<Mode, SequenceStrategy> 路由不同模式 |
| 服务状态 | ServiceState | RUNNING → SHUTTING_DOWN → SHUTDOWN 单向状态机 |

### 12.3 架构决策记录（ADR）

| 编号 | 决策 | 理由 | 替代方案 |
|------|------|------|----------|
| ADR-01 | 使用 mode (STRICT/CACHED) 替代 cache_size | 语义与 Oracle 一致，职责单一 | cache_size 同时表示模式和大小 |
| ADR-02 | 使用 REQUIRES_NEW 隔离序列事务 | 避免序列锁被业务操作延长 | REQUIRES（共享事务） |
| ADR-03 | 使用 AtomicLong 实现 Segment 内分配 | 无锁并发最高效 | synchronized（有锁） |
| ADR-04 | 不使用 Redis/Caffeine 外部缓存 | 减少依赖，部署简单 | 使用外部缓存提高容灾 |
| ADR-05 | 每次加载号段插入 sequence_record | 审计追踪需要 | 只更新 max_id 不记录 |
| ADR-06 | 同步加载作为兜底策略 | 保证极端情况下服务可用 | 仅依赖异步加载 |
| ADR-07 | 构造器注入 | 符合 Spring 最佳实践，便于测试 | @Autowired 字段注入 |
| ADR-08 | 动态步长限制 [100, 1000000] + 阻尼 2x | 避免步长过小或过大导致不稳定 | 不做限制 |
| ADR-09 | 使用 ConcurrentHashMap 缓存配置 | 无锁并发读，性能最优 | synchronized（有锁） |
| ADR-10 | ThreadLocal 实现会话语义 | 与 Oracle CURRVAL 语义一致 | 参数传递（破坏封装） |
| ADR-11 | TransactionTemplate 替代 @Transactional | 解决自调用 AOP 代理失效问题 | @Transactional（自调用失效） |
| ADR-12 | 移除心跳机制 | 不改变核心行为，减少 DB 写入开销 | 保留心跳（增加复杂度） |
| ADR-13 | 双重条件预加载 | 小步长场景下 10% 阈值窗口过小 | 仅使用率阈值 |
| ADR-14 | SequenceConfig 使用包装类型 + null-safe getter | 防止 MyBatis 映射 NULL 时拆箱 NPE | 原始类型（NPE 风险） |
| ADR-15 | 引入 SequenceStrategy 策略接口 | 支持 STRICT/CACHED/NATIVE 三种模式统一路由 | if-else 分支路由 |
| ADR-16 | ServiceState 枚举替代 volatile boolean | 显式化状态转换，防止非法转换 | 分散的 boolean 标志 |
| ADR-17 | EmptySegment Null Object 模式 | 消除 DoubleBuffer 中的 null 检查 | null 引用 + null 检查 |
| ADR-18 | SharedDataSourceWrapper fallback | 简化配置，未配置独立数据源时自动复用主数据源 | 显式配置独立数据源 |
