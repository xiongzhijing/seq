## 1. 项目概述

### 1.1 项目背景

Oracle 数据库提供原生的 Sequence 对象，支持通过 `NEXTVAL` 和 `CURRVAL` 获取严格连续的序列号，广泛应用于业务流水号、订单编号、单据序号等场景。MySQL 及国产 TDSQL（MySQL 兼容分支）未提供等价的 Sequence 机制，仅支持表级 `AUTO_INCREMENT`，无法满足以下需求：

- 多表共享同一序列
- 自定义步长、边界值、循环策略
- 跨实例严格连续递增

mysql-sequence 项目旨在填充这一空白，为 TDSQL/MySQL 环境下的 Spring Boot 应用提供 Oracle 兼容的 Sequence 能力。

### 1.2 项目目标

| 目标编号 | 目标描述 | 验收标准 |
|----------|----------|----------|
| BG-01 | 为 TDSQL + Spring Boot 环境提供 Oracle 兼容的 Sequence 能力 | 功能测试通过率 100% |
| BG-02 | 支持严格连续与高性能两种模式，供业务按需选择 | 严格连续模式 TPS ≥ 50（单实例）；号段缓存模式 TPS ≥ 5000 |
| BG-03 | 支持 TDSQL 原生 Sequence，消除应用层行锁 | `sequence.native-sequence-dialect=TDSQL` 激活后 STRICT 模式走 `tdsql_nextval()` |
| BG-04 | 数据库短时间不可用时，号段缓存模式仍能继续服务 | 模拟 DB 断开 5 分钟，服务不中断 |

### 1.3 项目范围

**范围内**：
- Spring Boot Starter 组件开发（jar 包方式交付）
- 严格连续模式（STRICT）：基于 `SELECT ... FOR UPDATE` 悲观锁
- 号段缓存模式（CACHED）：基于 Segment + 双 Buffer 机制
- 原生序列模式（NATIVE）：基于 TDSQL 原生 Sequence（`tdsql_nextval()`）
- 双 Buffer 异步预加载 + 动态步长调整
- 优雅关闭、持久化标记等宕机抵御机制
- currVal 会话语义（ThreadLocal 实现）
- 序列状态监控 REST 接口（默认禁用）
- 两张数据库表：sequence_config、sequence_record

**范围外**：
- Snowflake 算法（参考美团 Leaf 但明确不实现）
- 认证授权（auth）、告警（alerting）
- 分库分表（sharding）、多租户（multi-tenancy）
- 限流（rate limiting）
- 独立部署进程（本组件是库，嵌入宿主应用）

### 1.4 设计思想

本组件在借鉴美团 Leaf 号段模式核心思想的基础上，扩展支持 Oracle 风格的序列语义，并新增 TDSQL 原生 Sequence 支持：

| 借鉴来源 | 借鉴内容 | 扩展内容 |
|----------|----------|----------|
| 美团 Leaf | 号段模式（Segment） | Oracle CYCLE / START WITH 语义 |
| 美团 Leaf | 双 Buffer 机制 | 严格连续模式（FOR UPDATE） |
| 美团 Leaf | 动态步长调整 | currVal 会话语义 |
| Oracle Sequence | NEXTVAL / CURRVAL | 事务隔离 REQUIRES_NEW |
| TDSQL | 原生 Sequence（tdsql_nextval） | NativeSequenceDialect 抽象层 + 懒创建自愈 |

**部署模型的关键特征**：

| 特征 | 说明 |
|------|------|
| **实例 = Spring Boot 应用** | 通常是 K8s Pod 或 JVM 进程，不是独立序列服务进程 |
| **实例 ID = ip:port** | 用于 sequence_record 中的审计追踪，标识哪个实例分配了哪个号段 |

| **多实例协调** | 通过数据库行锁（非 RPC）实现跨实例串行化 |
| **生命周期** | Bean 随 Spring 上下文启动/关闭，通过 `ContextClosedEvent` 感知关闭 |

## 3. 核心组件设计

### 3.1 SequenceService（统一入口）

**类路径**：`com.ccb.jx.seq.core.SequenceService`

**职责**：
- 作为对外统一 API 门面
- 根据序列配置中的 `mode` 字段通过策略映射路由到对应的 `SequenceStrategy` 实现
- 维护配置缓存（ConcurrentHashMap），缓存序列配置对象
- 支持配置懒创建（Level 1）：当 `sequence_config` 表不存在对应行时，自动 INSERT 默认配置
- 定时刷新配置缓存，感知 DBA 配置变更
- 实现 `nextVal()` 和 `currVal()` 两个核心接口

**类设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                    SequenceService                           │
├─────────────────────────────────────────────────────────────┤
│ - strategyMap: Map<Mode, SequenceStrategy>                  │
│ - sequenceRepository: SequenceRepository                    │
│ - configCache: ConcurrentMap<String, SequenceConfig>        │
│ - cacheRefreshScheduler: ScheduledExecutorService           │
│ - configCacheTtlMinutes: long                               │
├─────────────────────────────────────────────────────────────┤
│ + nextVal(seqName: String): long                            │
│ + currVal(seqName: String): long                            │

│ + getAllConfigs(): List<SequenceConfig>                     │

**模式路由逻辑**：

```

   │
   ▼
[seqName 非空校验] ── 非法 ──► [抛出 NullPointerException]
   │ 合法
   ▼
[从缓存查询 SequenceConfig]
   │
   ├── 缓存未命中 ──► [从数据库加载配置]
   │                     │
   │                     ├── 配置不存在 ──► [自动 INSERT 默认配置(Level 1 懒创建)]
   │                     │                   → 默认 mode=STRICT
   │                     │
   │                     └── [写入缓存]
   │
   ▼
[strategyMap.get(config.getMode())]
   │
   ├── mode = STRICT + dialect=NONE ──► [StrictSequence.nextVal(seqName, config)]
   │                                        │
   │                                        ▼
   │                                   ┌─────────────────────────────┐
   │                                   │ TransactionTemplate          │
   │                                   │ (PROPAGATION_REQUIRES_NEW,   │
   │                                   │  timeout=3s)                 │

   │                                   │ 计算下一个值                   │
   │                                   │ UPDATE max_id                │

**核心代码设计**：

```java
public class SequenceService {

    private final Map<Mode, SequenceStrategy> strategyMap;
    private final SequenceRepository sequenceRepository;
    private final ConcurrentMap<String, SequenceConfig> configCache;
    private final ScheduledExecutorService cacheRefreshScheduler;
    private final long configCacheTtlMinutes;

    public SequenceService(Map<Mode, SequenceStrategy> strategyMap,
                           SequenceRepository sequenceRepository,
                           long configCacheTtlMinutes) {
        this.strategyMap = Collections.unmodifiableMap(
                Objects.requireNonNull(strategyMap, "strategyMap must not be null"));
        this.sequenceRepository = Objects.requireNonNull(sequenceRepository);
        this.configCacheTtlMinutes = configCacheTtlMinutes;
        this.configCache = new ConcurrentHashMap<>();
        this.cacheRefreshScheduler = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "sequence-config-cache-refresh");
            t.setDaemon(true);
            return t;
        });
        ConfigCacheRefresher refresher = new ConfigCacheRefresher(
                getCachedSequence(), sequenceRepository, configCache);
        cacheRefreshScheduler.scheduleAtFixedRate(
                refresher, configCacheTtlMinutes, configCacheTtlMinutes, TimeUnit.MINUTES);
    }

    public long nextVal(String seqName) {
        SequenceConfig config = getConfig(seqName);
        SequenceStrategy strategy = strategyMap.get(config.getMode());
        if (strategy == null) {
            throw new SequenceException(SequenceErrorCode.INVALID_CONFIG,
                    "Unsupported mode: " + config.getMode());
        }
        long value = strategy.nextVal(seqName, config);
        SessionHolder.setCurrVal(seqName, value);
        return value;
    }

    public long currVal(String seqName) {
        return SessionHolder.getCurrVal(seqName);
    }
}
```

### 3.2 StrictSequence（严格连续模式）

**类路径**：`com.ccb.jx.seq.core.StrictSequence`

**职责**：
- 通过 `SELECT ... FOR UPDATE` 行级悲观锁实现多实例串行化
- 每次 `nextVal` 在独立事务（TransactionTemplate + REQUIRES_NEW, timeout=3s）中完成"读取 → 计算 → 更新"三步操作
- 完整支持 Oracle 语义：increment_by、min_value、max_value、cycle
- 支持 start_with 初始化修正

**激活条件**：`sequence.native-sequence-dialect=NONE`（默认）或未配置

**类设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                    StrictSequence                            │
├─────────────────────────────────────────────────────────────┤
│ - sequenceRepository: SequenceRepository                    │
│ - transactionTemplate: TransactionTemplate                  │
├─────────────────────────────────────────────────────────────┤
│ + nextVal(seqName: String, config: SequenceConfig): long    │
│ - nextValInternal(seqName: String, config: SequenceConfig): │
│     long                                                    │
└─────────────────────────────────────────────────────────────┘
```

**nextVal 内部流程**：

```
┌─────────────────────────────────────────────────────────────┐
│              StrictSequence.nextVal 内部流程                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [TransactionTemplate(PROPAGATION_REQUIRES_NEW, timeout=3s)]│
│    │                                                        │
│    ▼                                                        │
│  [SELECT max_id, increment_by, min_value, max_value,        │
│   cycle, start_with FOR UPDATE]                             │
│    │                                                        │
│    ▼                                                        │
│  [start_with 初始化修正]                                     │
│    │ 当 max_id 为 DDL 默认值（1）且 start_with > 1 时，      │
│    │ 修正 currentMaxId = startWith - incrementBy             │
│    │ 使得 nextVal = (startWith - incrementBy) + incrementBy  │
│    │       = startWith                                      │
│    ▼                                                        │
│  [计算下一个值]                                              │
│    │                                                        │
│    ├── candidate = currentMaxId + increment_by               │
│    │                                                        │
│    ├── [检查是否超过 max_value]                                │
│    │    ├── 超过 + cycle=1  ──► [回绕到 min_value]           │
│    │    │                        newMaxId = minValue         │
│    │    ├── 超过 + cycle=0  ──► [抛出 SEQ_EXHAUSTED]        │
│    │    └── 未超过 ──► [正常递增, newMaxId = nextVal]        │
│    │                                                        │
│    ▼                                                        │
│  [UPDATE sequence_config SET max_id = newMaxId               │
│   WHERE seq_name = ? AND max_id = originalMaxId]             │
│    │ CAS 安全网（FOR UPDATE 已保证串行化，CAS 作为额外保障）   │
│    ▼                                                        │
│  [COMMIT (事务提交, 释放锁)]                                  │
│    │                                                        │
│    ▼                                                        │
│  [返回 nextVal]                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Oracle 语义实现**：

| Oracle 特性 | 映射实现 |
|-------------|----------|
| `NEXTVAL` | `nextVal(seqName)` 统一入口 |
| `INCREMENT BY n` | `sequence_config.increment_by` 字段，每次递增 n |
| `START WITH n` | `sequence_config.start_with` 字段，首次分配从此值开始 |
| `MAXVALUE n` | `sequence_config.max_value` 字段，达到后检查 |
| `MINVALUE n` | `sequence_config.min_value` 字段，循环回绕目标值 |
| `CYCLE` | `sequence_config.cycle` 字段，达到 max_value 后回绕 |
| `NOCYCLE` | cycle = 0，达到 max_value 后抛出异常 |
| `CURRVAL` | `currVal(seqName)` 会话级当前值 |

### 3.4 NativeStrictSequence（原生序列模式）

**类路径**：`com.ccb.jx.seq.core.NativeStrictSequence`

└─────────────────────────────────────────────────────────────┘
```

**nextVal 内部流程**：

```

│           NativeStrictSequence.nextVal 内部流程              │
├─────────────────────────────────────────────────────────────┤

│  [检查序列是否耗尽]                                           │

│    └── 是 ──► [抛出 SEQ_EXHAUSTED]                          │

### 3.5 DoubleBuffer（双 Buffer 机制）

**类路径**：`com.ccb.jx.seq.buffer.DoubleBuffer`

**职责**：
- 管理两个 Segment 引用（current 当前使用 + standby 备用），使用 EmptySegment 替代 null
- 实现零等待切换：当前 Segment 耗尽时立即切换到备用
- 触发异步预加载：双条件触发（90% 使用率 OR 剩余数量阈值）
- 维护切换锁，保证切换操作的原子性
- 提供活循环监控（staleResetCount）
- 延迟初始化：首次 `next()` 时同步加载第一个 Segment

**类设计**：

```

│ - preloadTriggered: AtomicBoolean    (预加载触发标记)        │

├─────────────────────────────────────────────────────────────┤
│ + next(): long                        (分配下一个值)         │

│ + remaining(): long                   (总剩余数量)           │

│ + getCurrent(): Segment               (获取当前号段)         │
│ + getStandby(): Segment               (获取备用号段)        │
│ + setAsyncTaskSubmitter(Runnable): void (设置异步提交器)     │
│ + getStaleResetCount(): int           (获取活循环监控计数)   │
└─────────────────────────────────────────────────────────────┘
```

**双 Buffer 状态机**：

```
                     ┌─────────────────┐
                     │   INIT (初始化)   │
                     │  current=Empty   │
                     │  standby=Empty   │
                     └────────┬─────────┘
                              │ 首次 next() 触发同步加载
                              ▼
                     ┌─────────────────┐
                     │  SINGLE (单Buffer) │
                     │  current=[1,1000] │
                     │  standby=Empty   │
                     └────────┬─────────┘
                              │ 到达预加载阈值 → 异步加载
                              ▼
          ┌─────────────────────────────────────┐
          │         DUAL (双Buffer)              │
          │  current=[1,1000], standby=[1001,2000] │
          └────────┬─────────────┬──────────────┘
                   │             │
          current 耗尽     current 未耗尽
                   │             │
                   ▼             ▼
          ┌──────────────┐  ┌──────────────────┐
          │ SWITCH (切换) │  │ NORMAL (正常分配) │
          │ current=standby│  │ 继续从 current   │
          │ standby=Empty  │  │ 分配             │
          └───────┬──────┘  └──────────────────┘
                  │
                  ▼
          ┌─────────────────┐
          │  SINGLE (单Buffer) │ ====> 循环回到 DUAL
          └─────────────────┘
```

**双条件预加载触发**：

预加载判断使用两个条件 OR 组合，任一满足即触发：

| 条件 | 公式 | 说明 |
|------|------|------|
| 条件①：使用率阈值 | `usage >= (1 - PRELOAD_THRESHOLD)` | 默认 90% 使用率时触发 |
| 条件②：剩余数量阈值 | `remaining < max(50, totalSize × 0.3)` | 解决小步长场景下条件①窗口过小的问题 |

**零等待切换机制**：

```
时间轴    nextVal线程A     nextVal线程B     异步预加载线程
  │          │                │                 │
  │    [从current分配]   [从current分配]        │
  │    剩余 5/1000      剩余 4/1000            │
  │          │                │                 │
  │    [剩余=0]          [剩余=0]              │
  │    nextVal返回        nextVal返回           │
  │    最后值 1000       发现耗尽               │
  │          │                │                 │
  │          │          [获取ReentrantLock]     │
  │          │                │                 │
  │    [等待锁]        [检查standby]            │
  │          │           standby.isInitialized()│
  │          │           ✅                     │
  │          │                │                 │
  │          │          [切换: current=standby]  │
  │          │          [standby=EmptySegment]   │
  │          │          [重置 preloadTriggered]  │
  │          │          [释放锁]                │
  │          │                │                 │
  │    [获取锁成功]     [从新current分配]        │
  │    [检查current]     返回 1001               │
  │    current已切换                              │
  │    [从新current分配]  ←── 或 ────►  [异步加载]│
  │          │                │                 │
  └─────────────────────────────────────────────┘
```

**关键特性**：
- **零等待**：切换操作在 ReentrantLock 保护下毫秒级完成
- **预加载异步化**：预加载在独立线程池执行，不阻塞分配线程
- **降级处理**：备用不可用时同步加载，保证服务不中断
- **活循环防护**：staleResetCount 监控异步任务被线程池丢弃后的标记重置
- **EmptySegment**：使用 Null Object 模式替代 null 引用，消除空指针检查

### 3.6 Segment（号段模型）

**类路径**：`com.ccb.jx.seq.buffer.Segment`

**职责**：
- 表示从数据库预取的一段连续序列号范围 [start, end]
- 使用 AtomicLong 提供线程安全的号段分配
- 提供剩余数量查询，供 DoubleBuffer 判断预加载时机
- 步长固定为 1（`AtomicLong.getAndIncrement()`），不支持 `incrementBy > 1` 的跨步分配

**类设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                       Segment                                │
├─────────────────────────────────────────────────────────────┤
│ - start: long                       (号段起始值, 包含)       │
│ - end: long                         (号段结束值, 包含)       │
│ - current: AtomicLong               (当前分配位置)           │
│ - initialized: volatile boolean      (是否已初始化)           │
│ - loadTime: volatile long           (号段加载时间戳)         │
├─────────────────────────────────────────────────────────────┤
│ + Segment(start: long, end: long)                            │
│ # Segment(start: long, end: long, skipValidation: boolean)  │
│ + next(): long                      (获取下一个值)           │
│ + remaining(): long                 (剩余可用数量)           │
│ + isExhausted(): boolean            (是否耗尽)               │
│ + usage(): double                   (使用率 0.0~1.0)        │
│ + used(): long                      (已使用数量)             │
│ + isInitialized(): boolean          (是否已初始化)           │
│ + getLoadTime(): long               (获取加载时间戳)         │
└─────────────────────────────────────────────────────────────┘
```

**内存数据模型**：

```
Segment 对象在内存中的状态表示:

  start = 1001          end = 2000
      │                    │
      ▼                    ▼
  ┌────┬──────────────────────────────────────┬────┐
  │    │                                      │    │
  │ 1001  ● ● ● ● ● ● ● ● ● ● ● ● ● ● ●  2000  │
  │        ▲                                    │
  │        │                                    │
  │   current = 1500                            │
  │   (AtomicLong)                              │
  │                                            │
  │   remaining = end - current + 1             │
  │            = 2000 - 1500 + 1 = 501          │
  │                                            │
  │   usage = (current - start) / (end - start + 1) │
  │         = (1500 - 1001) / 1000 ≈ 50%      │
  └────────────────────────────────────────────┘
```

**线程安全策略**：

- `next()` 使用 `AtomicLong.getAndIncrement()` 实现无锁并发分配
- 超出 `end` 时返回 -1，但 `current` 可能被推到 `end + 1` 以上，不影响正确性
- `remaining()` 和 `usage()` 为近似值，用于监控和预加载判断
- CACHED 模式下，Segment 内步长固定为 1，`increment_by` 配置不影响 Segment 分配

### 3.7 EmptySegment（空号段）

**类路径**：`com.ccb.jx.seq.buffer.EmptySegment`

**职责**：
- Null Object 模式，替代 DoubleBuffer 中的 null 引用
- 表示尚未加载号段的状态，所有操作均返回"已耗尽"语义
- 消除 DoubleBuffer 中的 null 检查

**类设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                     EmptySegment                             │
├─────────────────────────────────────────────────────────────┤
│ + INSTANCE: EmptySegment (全局唯一实例)                       │
├─────────────────────────────────────────────────────────────┤
│ - EmptySegment()  → super(0, -1, true)                      │
│ + next(): long          → -1 (号段耗尽)                      │
│ + getCurrent(): long    → -1                                 │
│ + remaining(): long     → 0                                  │
│ + used(): long          → 0                                  │
│ + isExhausted(): boolean → true                              │
│ + usage(): double       → 1.0                                │
│ + isInitialized(): boolean → false                           │
│ + getLoadTime(): long   → 0                                  │
└─────────────────────────────────────────────────────────────┘
```

**关键设计**：
- 使用 3 参数构造器 `super(0, -1, true)` 跳过参数校验和 AtomicLong 初始化
- `isInitialized()` 返回 `false`，用于 DoubleBuffer 判断 standby 是否可用
- `usage()` 返回 `1.0`（100% 使用率），使 `needsPreload()` 对 EmptySegment 返回 `false`

### 3.8 SequenceStrategy（策略接口）

**类路径**：`com.ccb.jx.seq.core.SequenceStrategy`

**职责**：
- 定义统一的序列生成策略接口
- 通过 `SequenceService` 的 `strategyMap`（`Map<Mode, SequenceStrategy>`）实现模式路由
- 三个实现：StrictSequence、CachedSequence、NativeStrictSequence

**接口定义**：

```java
@FunctionalInterface
public interface SequenceStrategy {
    long nextVal(String seqName, SequenceConfig config);
}
```

**策略映射**：

| Mode | SequenceStrategy 实现 | 激活条件 |
|------|----------------------|----------|
| STRICT | StrictSequence | `native-sequence-dialect=NONE`（默认） |
| STRICT | NativeStrictSequence | `native-sequence-dialect=TDSQL` |
| CACHED | CachedSequence | 始终存在 |

**路由机制**：

```java
// SequenceAutoConfiguration 中构建 strategyMap
Map<Mode, SequenceStrategy> strategyMap = new HashMap<>();
strategyMap.put(Mode.STRICT, strictStrategy);  // StrictSequence 或 NativeStrictSequence
strategyMap.put(Mode.CACHED, cachedSequence);

// SequenceService.nextVal 中路由
SequenceStrategy strategy = strategyMap.get(config.getMode());
long value = strategy.nextVal(seqName, config);
```

### 3.9 ServiceState（服务状态枚举）

**类路径**：`com.ccb.jx.seq.core.ServiceState`

**职责**：
- 显式化服务状态转换，替代分散的 boolean 标志
- 状态转换规则：`RUNNING → SHUTTING_DOWN → SHUTDOWN`，单向不可逆

**枚举定义**：

```java
public enum ServiceState {
    RUNNING,        // 正常运行中，接受请求
    SHUTTING_DOWN,  // 关闭中，拒绝新请求
    SHUTDOWN;       // 已关闭

    public boolean acceptRequests() {
        return this == RUNNING;
    }

    public boolean canTransitionTo(ServiceState target) {
        switch (this) {
            case RUNNING:      return target == SHUTTING_DOWN;
            case SHUTTING_DOWN: return target == SHUTDOWN;
            default:           return false;
        }
    }
}
```

**状态转换图**：

```
  RUNNING ──────► SHUTTING_DOWN ──────► SHUTDOWN
  (接受请求)       (拒绝新请求)          (已关闭)
     │                  │                    │
     │ transitionTo()   │ transitionTo()     │
     │ 合法             │ 合法               │
     └──────────────────┴────────────────────┘
           其他转换均为非法，抛出 IllegalStateException
```

### 3.10 ConfigCacheRefresher（配置缓存刷新器）

**类路径**：`com.ccb.jx.seq.core.ConfigCacheRefresher`

**职责**：
- 由 `SequenceService` 的定时调度器周期性调用
- 执行三项维护操作：清理过期 Buffer → 清空配置缓存 → 删除旧记录
- 连续失败 3 次以上升级为 ERROR 日志

**类设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                  ConfigCacheRefresher                         │
├─────────────────────────────────────────────────────────────┤
│ - cachedSequence: CachedSequence (可能为 null)               │
│ - sequenceRepository: SequenceRepository                    │
│ - configCache: ConcurrentMap<String, SequenceConfig>        │
│ - refreshFailCount: AtomicInteger                           │
├─────────────────────────────────────────────────────────────┤
│ + run(): void                                               │
│ - cleanupStaleBuffers(): void                               │
│ - refreshConfigCache(): void                                │
│ - cleanupOldRecords(): void                                 │
│ + getRefreshFailCount(): int                                │
└─────────────────────────────────────────────────────────────┘
```

**三步刷新流程**：

```
┌─────────────────────────────────────────────────────────────┐
│              ConfigCacheRefresher.run() 执行流程             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [Step 1: cleanupStaleBuffers()]                            │
│    │ 查询数据库中所有活跃序列名称                              │
│    │ 移除 bufferMap 中已不在配置表中的 DoubleBuffer           │
│    │ cachedSequence 可能为 null（检查后调用）                 │
│    ▼                                                        │
│  [Step 2: refreshConfigCache()]                             │
│    │ 粗粒度清空整个 configCache                              │
│    │ 下次访问时从数据库重新加载                               │
│    │ 确保 DBA 修改配置后能在 TTL 时间内生效                   │
│    ▼                                                        │
│  [Step 3: cleanupOldRecords()]                              │
│    │ 删除 sequence_record 中 alloc_time 超过 30 天的记录     │
│    │ 仅删除 EXPIRED/RECYCLED 状态的记录                      │
│    │ 此操作失败不影响主流程（catch 后仅 WARN）                │
│    ▼                                                        │
│  [重置 refreshFailCount = 0]                                │
│                                                             │
│  ──── 异常处理 ────                                         │
│  任一步骤抛出异常:                                           │
│    refreshFailCount.incrementAndGet()                       │
│    ≥ 3 次 → ERROR 日志                                      │
│    < 3 次 → WARN 日志                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.12 TdsqlSequenceDialect（TDSQL 方言实现）

**类路径**：`com.ccb.jx.seq.dialect.TdsqlSequenceDialect`

**职责**：
- 实现 TDSQL 特有的原生 Sequence SQL 语法
- 使用 `JdbcTemplate`（非 MyBatis）执行 SQL
- 处理 NOMAXVALUE 边界条件

**SQL 映射**：

| 操作 | SQL 语法 |
|------|----------|
| 获取值 | `SELECT tdsql_nextval(?)` |
| 创建序列 | `CREATE TDSQL_SEQUENCE {seqName} START WITH {startWith} TDSQL_MINVALUE {minValue} [TDSQL_NOMAXVALUE / TDSQL_MAXVALUE] TDSQL_INCREMENT BY {incrementBy} [TDSQL_CYCLE / TDSQL_NOCYCLE]` |
| 删除序列 | `DROP TDSQL_SEQUENCE {seqName}` |
| 检查存在 | `SELECT tdsql_lastval(?)`（异常 → 不存在） |

**NOMAXVALUE 处理**：

```java
// maxValue 接近 Long.MAX_VALUE 时使用 NOMAXVALUE
if (maxValue >= Long.MAX_VALUE - 1) {
    ddl.append(" TDSQL_NOMAXVALUE");
} else {
    ddl.append(" TDSQL_MAXVALUE ").append(maxValue);
}
```

**isSequenceNotExist 异常判断逻辑**：

遍历 cause 链，检查以下条件：

| 检查项 | 匹配内容 |
|--------|----------|
| 消息关键字 | "NOT EXIST"、"NOT EXISTS"、"1146"、"ER_NO_SUCH_TABLE" |
| SQLState | "42S02"（表/视图不存在）、"42S22"（列不存在） |

### 3.14 SharedDataSourceWrapper（共享数据源包装器）

**类路径**：`com.ccb.jx.seq.config.SequenceAutoConfiguration.SharedDataSourceWrapper`（内部类）

**职责**：
- Fallback 模式下包装应用的 DataSource
- 防止 Spring 容器关闭时误关闭应用数据源
- 继承 `AbstractDataSource`，不实现 `Closeable`

**关键设计**：

```java
static class SharedDataSourceWrapper extends AbstractDataSource {
    private final DataSource delegate;

    SharedDataSourceWrapper(DataSource delegate) {
        this.delegate = delegate;
    }

    @Override
    public Connection getConnection() throws SQLException {
        return delegate.getConnection();
    }

    @Override
    public Connection getConnection(String username, String password) throws SQLException {
        return delegate.getConnection(username, password);
    }
}
```

**为什么需要 SharedDataSourceWrapper**：
- 独立 HikariCP 数据源实现了 `Closeable`，Spring 会自动推断并调用 `close()`
- Fallback 模式下使用应用 DataSource，应用数据源应由应用自身管理生命周期
- 此包装器不实现 `Closeable`，Spring 不会为其推断 destroyMethod

## 6. 接口设计

### 6.1 内部服务接口

#### nextVal(seqName)

**签名**：`long nextVal(String seqName)`

**功能**：获取指定序列的下一个值。

**请求**：

| 参数 | 类型 | 说明 |
|------|------|------|
| seqName | String | 序列名称，对应 sequence_config.seq_name |

**响应**：

| 返回值 | 类型 | 说明 |
|--------|------|------|
| 序列值 | long | 下一个可用的序列号 |

**异常**：

| 异常场景 | 异常类型 | 错误码 |
|----------|----------|--------|
| 序列不存在 | SequenceException | SEQ_NOT_FOUND |
| 序列耗尽（非循环模式） | SequenceException | SEQ_EXHAUSTED |
| 关闭中拒绝服务 | SequenceException | SHUTTING_DOWN |
| 数据库操作失败 | SequenceException | DB_ERROR |
| 序列配置无效 | SequenceException | INVALID_CONFIG |
| 号段加载失败 | SequenceException | SEGMENT_LOAD_FAILED |
| 原生序列操作失败 | SequenceException | NATIVE_SEQ_ERROR |

**路由规则**：

| mode | dialect | 路由目标 | 机制 |
|------|---------|----------|------|
| STRICT | NONE | StrictSequence | `SELECT ... FOR UPDATE` |
| STRICT | TDSQL | NativeStrictSequence | `tdsql_nextval()` |
| CACHED | - | CachedSequence | Segment + DoubleBuffer |

**使用示例**：

```java
@Service
public class OrderService {

    private final SequenceService sequenceService;

    public OrderService(SequenceService sequenceService) {
        this.sequenceService = sequenceService;
    }

    public void createOrder() {
        long orderNo = sequenceService.nextVal("ORDER_NO");
        // 使用 orderNo 创建订单
    }
}
```

#### currVal(seqName)

**签名**：`long currVal(String seqName)`

**功能**：获取当前会话中最后一次 `nextVal` 的值。会话定义为一次 HTTP 请求的生命周期，通过 ThreadLocal 实现隔离。

**请求**：

| 参数 | 类型 | 说明 |
|------|------|------|
| seqName | String | 序列名称 |

**响应**：

## 7. 安全设计

### 7.1 事务隔离设计

#### 7.1.1 核心原则：序列事务与业务事务完全解耦

```
┌─────────────────────────────────────────────────────────────┐
│                   事务隔离总览                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  业务服务                   序列服务                          │
│  ┌────────────────┐        ┌────────────────────────────┐  │
│  │ @Transactional  │        │ TransactionTemplate         │  │
│  │ (业务事务)       │        │ (PROPAGATION_REQUIRES_NEW, │  │
│  │                │        │  timeout=3s)               │  │
│  │ BEGIN TX       │        │                            │  │
│  │ INSERT order   │        │ 独立事务上下文              │  │
│  │ INSERT item    │        │ SELECT ... FOR UPDATE      │  │
│  │ UPDATE inv     │        │ UPDATE ...                 │  │
│  │ COMMIT/ROLLBACK│        │ COMMIT (立即提交)           │  │
│  └────────────────┘        └────────────────────────────┘  │
│         │                            │                     │
│         └──────────── 解耦 ──────────┘                     │
│                                                             │
│  序列事务提交不依赖于业务事务的成功与否                        │
│  业务事务回滚不回滚序列事务（接受跳号）                        │
│                                                             │
│  使用 TransactionTemplate 而非 @Transactional 的原因：       │
│  自调用场景下 Spring AOP 代理不生效，编程式事务更可靠          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 7.1.2 StrictSequence 事务策略

```
┌─────────────────────────────────────────────────────────────┐
│           StrictSequence 事务策略                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  每次 nextVal 调用 = 一个独立事务                             │
│                                                             │
│  时间线:                                                     │
│    │                                                        │
│    ├── 业务线程上下文                                        │
│    │   ┌───────────────────────────────────────────────┐   │
│    │   │ 业务事务（若有）被挂起（REQUIRES_NEW）          │   │
│    │   └───────────────────────────────────────────────┘   │
│    │                                                        │
│    ├── 序列事务（TransactionTemplate 新上下文）              │
│    │   │                                                    │
│    │   ├── transactionTemplate.execute(...)                 │
│    │   ├── BEGIN TRANSACTION (timeout=3s)                  │
│    │   ├── SELECT max_id FROM sequence_config               │
│    │   │   WHERE seq_name = ? FOR UPDATE  ← 行级悲观锁     │
│    │   ├── start_with 初始化修正（如需要）                   │
│    │   ├── 计算 nextVal                                     │
│    │   ├── UPDATE sequence_config                            │
│    │   │   SET max_id = ?                                   │
│    │   │   WHERE seq_name = ? AND max_id = originalMaxId    │
│    │   ├── COMMIT  ← 锁释放                                 │
│    │   │                                                    │
│    │   └── 序列事务结束，业务事务恢复                          │
│    │                                                        │
│    └── 耗时: ~5ms（锁持有时间极短）                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 7.1.3 CachedSequence 事务策略

| 操作 | 是否需要事务 | 事务传播行为 | 说明 |
|------|-------------|-------------|------|
| nextVal 分配 | **不需要** | - | 纯内存 AtomicLong 操作 |
| 号段加载（loadSegment） | **需要** | REQUIRES_NEW | SELECT FOR UPDATE + UPDATE + INSERT，独立提交 |
| 异步预加载 | **需要** | REQUIRES_NEW | 在异步线程中通过 TransactionTemplate 执行 |

```
┌─────────────────────────────────────────────────────────────┐
│           CachedSequence 事务策略                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  主线程（nextVal 调用者）:                                    │
│    ┌──────────────────────────────────────────────────┐     │
│    │ 无需事务，纯内存操作                                │     │
│    │ nextVal = current.getAndIncrement()              │     │
│    │ 耗时: < 1μs                                       │     │
│    └──────────────────────────────────────────────────┘     │
│                                                             │
│  异步线程（loadSegment）:                                     │
│    ┌──────────────────────────────────────────────────┐     │
│    │ TransactionTemplate(PROPAGATION_REQUIRES_NEW,     │     │
│    │                     timeout=3s)                   │     │
│    │ BEGIN TX                                         │     │
│    │ SELECT ... FOR UPDATE                            │     │
│    │ 计算动态步长（含阻尼）                              │     │
│    │ UPDATE sequence_config                           │     │
│    │   SET max_id = newMaxId                          │     │
│    │   WHERE seq_name = ? AND max_id = oldMaxId       │     │
│    │                                                  │     │
│    │ INSERT INTO sequence_record (...)                 │     │
│    │ COMMIT                                           │     │
│    └──────────────────────────────────────────────────┘     │
│                                                             │
│  使用 TransactionTemplate 而非 @Transactional 的原因：      │
│  loadSegment 通过内部 lambda 被 DoubleBuffer 调用（自调用），  │
│  Spring AOP 代理不会拦截自调用，导致 @Transactional 失效。    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤
│                                                             │

### 7.2 锁策略

| 场景 | 锁机制 | 粒度 | 持有时间 |
|------|--------|------|----------|

#### 8.1.2 资源释放策略

| 资源类型 | 释放策略 | 超时时间 |
|----------|----------|----------|
| 异步线程池 | 调用 `shutdown()` → `awaitTermination()` → `shutdownNow()` | 5s |
| 配置缓存调度器 | 调用 `shutdown()` → `awaitTermination()` → `shutdownNow()` | 3s |
| 内存号段 | 记录日志，不强制回写数据库 | - |
| ThreadLocal | 由 SessionHolderCleanupInterceptor 自动清理 | 请求结束 |

#### 8.1.3 关闭期间请求处理策略

| 请求时机 | 处理策略 |
|----------|----------|
| 关闭前已提交的请求 | 等待完成（最多 5s） |
| 关闭过程中新到的请求 | 快速失败，抛出 SequenceException(SHUTTING_DOWN) |
| 异步预加载中的任务 | 等待当前任务完成，不提交新任务 |

### 8.2 持久化标记机制

#### 8.2.1 标记策略

每次从数据库加载新号段时（仅 CACHED 模式），在 `sequence_record` 表中插入一条标记记录，记录号段范围、分配实例和时间戳。该标记用于：

- **审计追溯**：记录所有号段分配历史，便于问题排查和运营分析
- **号段生命周期追踪**：记录号段从分配到过期/回收的完整状态流转

**注意**：号段丢失（gap）是 CACHED 模式下实例崩溃的预期行为，与 Oracle 行为一致。`sequence_record` 主要用于审计追踪，不承担宕机恢复职责。

## 9. 性能设计

### 9.1 动态步长调整

#### 9.1.1 算法公式

动态步长调整采用美团 Leaf 线性算法，根据号段实际消耗速率自动调整步长，使数据库更新周期稳定在目标值附近。同时引入阻尼机制，防止步长振荡。

```
nextStep = step × (SEGMENT_DURATION / actualDuration)

其中:
  SEGMENT_DURATION = 15 分钟 (目标更新时间间隔)
  actualDuration    = 上次号段从开始使用到用完的实际耗时（最小 1 分钟）
  step              = 当前步长
  nextStep          = 计算出的下一步长

阻尼约束: 单次调整不超过 2 倍
  maxStep = currentStep × 2
  minStep = ceil(currentStep / 2)
  nextStep = max(minStep, min(maxStep, nextStep))

全局约束范围: [100, 1000000]
```

#### 9.1.2 调整策略

| 场景 | actualDuration | 原始比例 | 阻尼后 nextStep 变化 | 含义 |
|------|---------------|----------|----------------------|------|
| 消耗过快 | 5 min | step × 3 | step × 2（受阻尼上限） | 步长最多扩大 2 倍 |
| 消耗适中 | 15 min | step × 1 | step × 1 | 步长保持不变 |
| 消耗较慢 | 30 min | step × 0.5 | step × 0.5 | 步长缩小一半 |
| 消耗很慢 | 60 min | step × 0.25 | ceil(step / 2)（受阻尼下限） | 步长最多缩小一半 |

#### 9.1.3 调整流程

```
┌─────────────────────────────────────────────────────────────┐
│                   动态步长调整流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [异步预加载完成 - 新号段加载成功]                            │
│    │                                                        │
│    ▼                                                        │
│  [获取当前 Segment 的 loadTime]                              │
│    │                                                        │
│    ▼                                                        │
│  [计算 actualDuration = now - loadTime]                     │
│    │ actualDuration 最小为 1 分钟                            │
│    │                                                        │
│    ▼                                                        │
│  [计算原始 nextStep]                                         │
│    │ nextStep = step × (SEGMENT_DURATION / actualDuration)  │
│    │                                                        │
│    ▼                                                        │
│  [应用阻尼约束]                                               │
│    │ maxStep = currentStep × 2                              │
│    │ minStep = ceil(currentStep / 2)                        │
│    │ nextStep = max(minStep, min(maxStep, nextStep))       │
│    │                                                        │
│    ▼                                                        │
│  [检查全局边界]                                               │
│    │ nextStep = max(100, min(1000000, nextStep))            │
│    │                                                        │

└─────────────────────────────────────────────────────────────┘
```

#### 9.1.4 实际效果模拟

假设初始 step = 1000，目标更新间隔 = 15min：

```
时间线        实际消耗时间   原始计算步长   阻尼后步长   号段范围             更新间隔
──────────────────────────────────────────────────────────────────────────────────
T0            -             1000         1000        [1, 1000]            -
T0+5min       5 min         3000         2000        [1001, 3000]         5 min (过快, 受阻尼)
T0+10min      5 min         6000         4000        [3001, 7000]         5 min (受阻尼)

稳定后         ~15 min       适度值       适度值       ...                 15 min (稳定)
```

**收敛特性**：阻尼机制使步长变化更平滑，避免振荡。经过 5-8 个周期后，步长自动收敛到使更新间隔稳定在 15min 的水平。

### 9.2 双 Buffer 零切换

#### 9.2.1 性能模型

```
┌─────────────────────────────────────────────────────────────┐
│               双 Buffer 性能模型                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  正常分配路径 (99.9% 请求):                                  │
│    ┌─────────────────────────────────────────────────┐      │
│    │ next() → AtomicLong.getAndIncrement() → 返回    │      │
│    │ 耗时: < 1μs (纳秒级)                            │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
│  切换路径 (0.1% 请求):                                       │
│    ┌─────────────────────────────────────────────────┐      │
│    │ next() → 发现耗尽 → ReentrantLock.lock()        │      │
│    │ → current = standby → unlock() → 返回           │      │
│    │ 耗时: < 100μs (微秒级)                          │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
│  异步预加载路径 (与分配路径完全并行):                         │
│    ┌─────────────────────────────────────────────────┐      │
│    │ 独立线程: SELECT FOR UPDATE → 计算动态步长       │      │
│    │ → UPDATE max_id → INSERT record → 构建 Segment  │      │
│    │ → 设置 standby Buffer                           │      │
│    │ 耗时: ~10ms (毫秒级，不影响分配线程)             │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 9.2.2 TP999 延迟保证

| 阶段 | 延迟 | 说明 |
|------|------|------|
| 正常分配 | < 1μs | AtomicLong CAS 操作 |
| 切换分配 | < 100μs | ReentrantLock 竞争 |
| 同步加载（兜底） | ~10ms | 仅当备用未就绪时触发 |
| TP999 | < 1ms | 99.9% 请求走正常分配路径 |

#### 9.2.3 预加载触发时机

```
预加载触发条件 (双条件 OR):

条件①: usage >= (1 - PRELOAD_THRESHOLD)
  → 使用率达到 90% 时触发

条件②: remaining < max(50, totalSize × 0.3)
  → 解决小步长场景下条件①窗口过小的问题

触发示例:
  step = 1000, threshold = 10%
  → 条件①: 剩余 < 100 时触发
  → 条件②: 剩余 < max(50, 300) = 300 时触发
  → 实际触发: 剩余 < 300 时（条件②先满足）
  → 预加载耗时 ~10ms，消耗 300 个号需 ~60ms (5000 TPS)
  → 预加载远快于消耗速度，100% 覆盖
```

| 优雅关闭 | 不支持 | 支持（ContextClosedEvent + @PreDestroy 双重入口） |
| 部署方式 | 独立服务 | 嵌入宿主应用（jar Starter） |
| 数据源隔离 | 不支持 | 支持（独立 DataSource + Fallback） |

### 11.4 架构决策记录（ADR）

| 编号 | 决策 | 理由 | 替代方案 |
|------|------|------|----------|
| ADR-01 | 使用 mode (STRICT/CACHED) 替代 cache_size | 语义与 Oracle 一致，职责单一 | cache_size 同时表示模式和大小 |
| ADR-02 | 使用 REQUIRES_NEW 隔离序列事务 | 避免序列锁被业务操作延长 | 使用 REQUIRES (共享事务) |
| ADR-03 | 使用 AtomicLong 实现 Segment 内分配 | 无锁并发最高效 | synchronized (有锁) |
| ADR-04 | 不使用 Redis/Caffeine 外部缓存 | 减少依赖，部署简单 | 使用外部缓存提高容灾 |
| ADR-05 | 每次加载号段都插入 sequence_record | 审计追踪需要分配记录 | 只更新 max_id 不记录 |
| ADR-06 | 同步加载作为兜底策略 | 保证极端情况下服务可用 | 仅依赖异步加载 |
| ADR-07 | 使用构造器注入 | 符合 Spring 最佳实践，便于测试 | @Autowired 字段注入 |
| ADR-08 | 动态步长限制 [100, 1000000] | 避免步长过小或过大导致不稳定 | 不做限制 |
| ADR-09 | 使用 TransactionTemplate 替代 @Transactional | 自调用场景下 AOP 代理不生效，编程式事务更可靠 | 保持 @Transactional 注解式 |
| ADR-10 | 引入 NativeSequenceDialect 抽象层 | 支持多种数据库原生序列，可扩展 | 直接硬编码 TDSQL 语法 |
| ADR-11 | SharedDataSourceWrapper 不实现 Closeable | 防止 Spring 容器关闭时误关闭应用 DataSource | 让 Spring 管理 DataSource 生命周期 |
| ADR-12 | EmptySegment 替代 null 引用 | Null Object 模式消除空指针检查，代码更简洁 | 使用 null 检查 |
| ADR-13 | ServiceState 枚举替代 boolean 标志 | 显式化状态转换，防止非法转换 | 分散的 boolean 标志 |
| ADR-14 | ConfigCacheRefresher 独立类 | 单一职责，从 SequenceService 中提取 | 内嵌在 SequenceService 中 |
| ADR-15 | 线程池使用 LinkedBlockingQueue(200) + CallerRunsPolicy | 有界队列防 OOM，CallerRunsPolicy 不丢弃任务 | SynchronousQueue + AbortPolicy |
| ADR-16 | NATIVE 模式无应用级事务 | 依赖 DB 原生序列原子性，max_id 更新为 best-effort | 为 max_id 更新引入事务 |
