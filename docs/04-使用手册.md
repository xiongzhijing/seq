## 1. 概述

### 1.1 项目定位

mysql-sequence 是一个 **Spring Boot Starter 库（Jar 包）**，**非独立服务**。它为 TDSQL / MySQL 环境提供 Oracle 兼容的 Sequence 能力，嵌入宿主应用的 Spring 上下文中运行。

### 1.2 核心特性

- **Oracle 兼容 API**：提供 `nextVal()` / `currVal()` 接口，语义与 Oracle Sequence 一致
- **三种模式**：STRICT（严格连续）、CACHED（号段缓存 + 双缓冲）、NATIVE（TDSQL 原生序列）
- **Spring Boot 自动配置**：引入依赖即用，支持独立数据源或回退到业务数据源
- **优雅关闭**：`ContextClosedEvent` + `@PreDestroy` 双入口保障，资源安全释放
- **会话隔离**：`ThreadLocal` 实现 Oracle CURRVAL 会话语义，HTTP 请求自动清理
- **懒创建**：首次 `nextVal` 时自动创建默认序列配置，无需预建表数据
- **动态步长**：根据消耗速率自动调整号段步长，稳定数据库更新周期

### 1.3 三种模式简介

| 模式 | 机制 | 适用场景 | TPS |
|------|------|----------|-----|
| STRICT | `SELECT ... FOR UPDATE` 行锁 | 财务凭证号、发票号等严格连续场景 | ~50（单机）~200（4实例） |
| CACHED | 号段预取 + 双缓冲 | 分库分表主键、日志ID等高吞吐场景 | ≥ 5,000（单机） |
| NATIVE | TDSQL 原生 `tdsql_nextval()` | TDSQL 环境高性能严格连续 | 取决于数据库 |

## 2. 快速开始

### 2.1 建表

在您的数据库中执行以下 DDL 建表语句。

```sql
-- 序列配置表
CREATE TABLE IF NOT EXISTS sequence_config (
    seq_name VARCHAR(64) NOT NULL PRIMARY KEY COMMENT '序列名称',
    max_id BIGINT NOT NULL DEFAULT 1 COMMENT '当前已分配的最大号段终点',
    step INT NOT NULL DEFAULT 1000 COMMENT '号段步长',
    increment_by INT NOT NULL DEFAULT 1 COMMENT '序列步长',
    min_value BIGINT NOT NULL DEFAULT 1 COMMENT '最小值',
    max_value BIGINT NOT NULL DEFAULT 9223372036854775807 COMMENT '最大值',
    cycle TINYINT NOT NULL DEFAULT 0 COMMENT '是否循环: 0-否, 1-是',
    mode VARCHAR(16) NOT NULL DEFAULT 'CACHED' COMMENT '序列模式: STRICT-严格连续, CACHED-号段缓存',
    start_with BIGINT NOT NULL DEFAULT 1 COMMENT '起始值(对应Oracle START WITH)',
    description VARCHAR(255) DEFAULT NULL COMMENT '描述',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='序列配置表';

-- 号段分配记录表（仅 CACHED 模式使用）
CREATE TABLE IF NOT EXISTS sequence_record (
    id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    seq_name VARCHAR(64) NOT NULL COMMENT '序列名称',
    start_value BIGINT NOT NULL COMMENT '号段起始值',
    end_value BIGINT NOT NULL COMMENT '号段结束值',
    instance_id VARCHAR(64) DEFAULT NULL COMMENT '应用实例ID',
    status VARCHAR(16) NOT NULL DEFAULT 'ALLOCATED' COMMENT '状态: ALLOCATED-已分配, EXPIRED-已过期, RECYCLED-已回收',
    alloc_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '分配时间',
    expire_time DATETIME DEFAULT NULL COMMENT '过期时间',
    INDEX idx_seq_name (seq_name),
    INDEX idx_instance_id (instance_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='序列号段分配记录';
```

### 2.2 引入依赖

在 `pom.xml` 中添加：

```xml
<dependency>
    <groupId>com.ccb.jx</groupId>
    <artifactId>mysql-sequence-spring-boot-starter</artifactId>
    <version>2.0.0</version>
</dependency>
```

> **注意**：确保您的项目中已引入 Spring Boot、MyBatis 和 JDBC 驱动（`mysql-connector-java`）。

### 2.3 配置数据源

推荐使用独立数据源，将序列操作的数据库连接与业务数据库隔离：

```yaml
sequence:
  datasource:
    url: jdbc:mysql://localhost:3306/sequence_db?useSSL=false&serverTimezone=Asia/Shanghai
    username: root
    password: root123
    driver-class-name: com.mysql.cj.jdbc.Driver
```

> `sequence.datasource.jdbc-url` 可作为 `url` 的别名（Spring Boot 2.x 兼容），二者优先级：`jdbc-url` > `url`。

**HikariCP 默认参数**（独立数据源模式下）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| maxPoolSize | 5 | 最大连接数 |
| minIdle | 1 | 最小空闲连接 |
| connectionTimeout | 3000ms | 连接超时 |
| maxLifetime | 1800000ms (30min) | 连接最大存活时间 |

#### 回退到业务数据源

如果**未配置** `sequence.datasource.url`，组件会自动回退（Fallback）到应用的主数据源：

```yaml
# 不配置 sequence.datasource.*，复用应用的 spring.datasource.*
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/business_db
    username: root
    password: root123
```

回退模式使用 `SharedDataSourceWrapper` 包装业务数据源，**不会**在容器关闭时误关闭业务连接池。

### 2.4 代码示例

以下是一个完整的 Spring Boot 示例，展示如何在业务代码中使用 Sequence：

```java
import com.ccb.jx.seq.core.SequenceService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class OrderService {

    @Autowired
    private SequenceService sequenceService;

    /**
     * 创建订单——使用 STRICT 模式保证订单号严格连续。
     * 首次调用 'ORDER_NO' 时，组件会自动创建默认配置（mode=STRICT, start_with=1）。
     */
    public String createOrder() {
        // 获取下一个订单号
        long orderNo = sequenceService.nextVal("ORDER_NO");

        // 在当前会话中获取上一次 nextVal 的值（Oracle CURRVAL 语义）
        long currentVal = sequenceService.currVal("ORDER_NO");

        // 业务逻辑...
        System.out.println("订单号: " + orderNo + ", 当前会话值: " + currentVal);
        return "ORDER-" + orderNo;
    }

    /**
     * 批量创建订单——使用 CACHED 模式高性能获取序列值。
     * 首次调用 'ORDER_ID' 时，会自动创建默认配置（mode=STRICT）。
     * 如需 CACHED 模式，需手动在 DB 中插入记录设置 mode=CACHED。
     */
    public void batchCreateOrders(int count) {
        for (int i = 0; i < count; i++) {
            long id = sequenceService.nextVal("ORDER_ID");
            // 业务逻辑...
        }
    }
}
```

> **首次调用会自动创建配置**：当 `sequence_config` 表中不存在对应的 `seq_name` 时，组件会自动 `INSERT IGNORE` 一条默认配置（mode=STRICT, step=1000, increment_by=1, start_with=1）。详见第 8 节"懒创建行为"。

### 3.3 回退机制说明

当 `sequence.datasource.url` 和 `sequence.datasource.jdbc-url` **均未配置**时：

1. 组件通过 `ObjectProvider<DataSource>` 获取 Spring 容器中的主数据源
2. 使用 `SharedDataSourceWrapper`（继承 `AbstractDataSource`，不实现 `Closeable`）包装
3. Spring 容器关闭时**不会**尝试关闭此包装器，业务数据源生命周期由应用自身管理
4. 独立 HikariCP 数据源实现了 `Closeable`，Spring 会自动推断并调用 `close()`

### 3.4 HikariCP 默认参数

仅独立数据源模式生效：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `maximumPoolSize` | 5 | 最大连接数 |
| `minimumIdle` | 1 | 最小空闲连接数 |
| `connectionTimeout` | 3000ms | 连接超时时间 |
| `maxLifetime` | 1,800,000ms (30min) | 连接最大存活时间 |
| `poolName` | `sequence-hikari-pool` | 连接池名称 |

## 4. 三种模式详解

### 4.1 STRICT（严格连续模式）

#### 机制

每次 `nextVal()` 调用在独立事务中执行：

1. `BEGIN TX`（`PROPAGATION_REQUIRES_NEW`, timeout=3s）
2. `SELECT ... FOR UPDATE` 锁定行记录，阻塞其他竞争实例
3. 读取当前 `max_id`，计算下一个值（含 `start_with` 修正）
4. `UPDATE sequence_config SET max_id = newValue`（FOR UPDATE 已保证串行化，无需 CAS）
5. `COMMIT`，释放行锁

#### 关键参数

| 参数 | 作用 |
|------|------|
| `increment_by` | **每次递增的步幅**，设为 2 则序列为 1,3,5,... |
| `start_with` | 起始值，如设为 10000 则序列从 10000 开始 |
| `min_value` | 最小值（循环模式下回绕到此值） |
| `max_value` | 最大值（达到后报错或循环） |
| `cycle` | 是否循环（0=报错，1=回绕到 min_value） |

> **注意**：`step` 字段在 STRICT 模式下**无意义**。只有 `increment_by` 控制递增步幅。

#### 性能

| 实例数 | TPS | 延迟 |
|--------|-----|------|
| 1 实例 | ~50 | ~20ms/次（行锁等待） |
| 4 实例 | ~100-200 | 取决于行锁竞争程度 |

#### 适用场景

- 财务凭证号（如 PZ-20240605-0001）
- 发票号（需严格连续、不可跳号）
- 订单号（业务要求严格递增）

---

### 4.2 CACHED（号段缓存模式）

#### 机制

采用双缓冲（DoubleBuffer）机制批量预取号段到内存：

1. **Segment 加载**：从 DB 批量获取一段连续范围（如 [1, 1000]）到内存
2. **内存分配**：通过 `AtomicLong.getAndIncrement()` 无锁分配
3. **双 Buffer 切换**：当前 Segment 即将耗尽时，切换到备用 Segment（零等待）
4. **异步预加载**：切换后立即异步加载新的备用 Segment

#### 双条件预加载

触发条件（OR 逻辑，任一满足即触发）：

- **条件①**：当前 Segment 使用率 ≥ 90%
- **条件②**：剩余数量 < `max(50, 总大小 × 0.3)` — 保障小步长场景有足够预加载窗口

#### 线程池配置

| 参数 | 值 | 说明 |
|------|-----|------|
| corePoolSize | 20 | 核心线程数 |
| maxPoolSize | 100 | 最大线程数 |
| keepAliveTime | 60s | 空闲线程存活时间 |
| workQueue | `LinkedBlockingQueue(200)` | 有界队列，容量 200 |
| RejectedExecutionHandler | `CallerRunsPolicy` | 拒绝时由调用线程执行 |

#### 动态步长调整

当消耗过快或过慢时，自动调整步长以稳定 DB 更新周期（目标 ~15 分钟）：

```
nextStep = round(currentStep × SEGMENT_DURATION / actualDuration)
```

- **阻尼机制**：单次调整不超过 2 倍（最小 `ceil(step/2)`，最大 `step×2`）
- **全局范围**：[100, 1,000,000]
- **最低保护**：`actualDuration` 最低 1 分钟，防止极端比例

#### 持久化标记

每次加载新号段时，在 `sequence_record` 表中 INSERT 一条记录：

| 字段 | 内容 |
|------|------|
| seq_name | 序列名称 |
| start_value | 号段起始值 |
| end_value | 号段结束值 |
| instance_id | 所属实例 ID |
| status | ALLOCATED |
| alloc_time | 分配时间戳 |

> **注意**：CACHED 模式下 `increment_by` 不生效，连续值之间的步长始终为 1。

#### 性能

| 指标 | 数值 |
|------|------|
| 单实例 TPS | ≥ 5,000 |
| TP999 延迟 | ≤ 1ms |
| DB 更新周期 | ~15-30 分钟（动态步长调整） |

#### 适用场景

- 分库分表全局主键
- 日志 ID、消息 ID
- 对跳号不敏感的高并发场景

---

### 4.3 NATIVE（原生序列模式）

#### 机制

通过 `sequence.native-sequence-dialect=TDSQL` 激活，将序列生成委托给数据库原生 Sequence：

1. 启动时 `NativeSequenceSynchronizer` 同步创建缺失的 DB Sequence
2. 运行时调用 `tdsql_nextval()` 生成值
3. `max_id` 更新采用 **best-effort** 策略：更新失败仅 WARN 日志，不阻塞返回值
4. 无应用层事务，依赖数据库原生原子性

#### 两层懒创建自愈

| 层级 | 对象 | 触发时机 | 机制 |
|------|------|----------|------|
| Level 1 | `sequence_config` 行 | 首次 `nextVal()` | `INSERT IGNORE` 默认配置 |
| Level 2 | DB Sequence 对象 | `dialect.nextVal()` 抛出"Sequence 不存在" | 自动 `CREATE` 并重试 |

#### 适用场景

- TDSQL 环境
- 需要严格连续 + 高吞吐
- 希望消除应用层行锁竞争

---

### 4.4 模式选择对比

| 对比维度 | STRICT | CACHED | NATIVE |
|----------|--------|--------|--------|
| **连续性** | 严格连续 | 趋势递增（同实例连续） | 严格连续 |
| **TPS** | ~100-200（4实例） | ≥ 5,000（单实例） | 取决于数据库 |
| **事务** | REQUIRES_NEW，每次提交 | 纯内存操作（加载时 REQUIRES_NEW） | 无应用级事务 |
| **宕机影响** | 最多丢 1 个值 | 可能丢整个号段 | 数据库原生保证 |
| **increment_by** | 生效 | 不生效（步长恒为 1） | 取决于方言实现 |
| **DB 依赖** | 实时依赖 | 号段加载时依赖 | 实时依赖 |
| **配置方式** | `mode=STRICT` | `mode=CACHED` | `mode=STRICT` + `native-sequence-dialect=TDSQL` |

## 5. API 参考

### 5.1 SequenceService

`SequenceService` 是序列服务的统一入口，通过构造器注入到业务 Bean 中使用。

#### nextVal

```java
/**
 * 获取下一个序列值。
 * 根据序列的 mode 自动路由到 STRICT / CACHED / NATIVE 实现。
 * 每次成功调用后自动记录 currVal 到 SessionHolder。
 *
 * @param seqName 序列名称，不能为 null 或空字符串
 * @return 下一个序列值
 * @throws SequenceException 如果序列不存在、已耗尽或服务正在关闭
 */
long nextVal(String seqName);
```

#### currVal

```java
/**
 * 获取当前会话中指定序列的最后一次 nextVal 返回值。
 * 对应 Oracle CURRVAL 语义。必须在同一线程中先调用 nextVal。
 *
 * @param seqName 序列名称，不能为 null
 * @return 当前会话中最后一次 nextVal 的值
 * @throws SequenceException 如果当前会话中尚未调用 nextVal
 */
long currVal(String seqName);
```

#### getConfig

```java
/**
 * 获取序列配置。
 * 优先从缓存读取；缓存未命中时从 DB 加载。
 * 若 DB 中不存在对应行，自动 INSERT IGNORE 默认配置。
 *
 * @param seqName 序列名称
 * @return 序列配置
 */
SequenceConfig getConfig(String seqName);
```

#### getAllConfigs

```java
/**
 * 获取所有序列配置（全量查询）。
 * 直接从数据库读取，不经过缓存。
 *
 * @return 全部序列配置列表
 */
List<SequenceConfig> getAllConfigs();
```

#### evictConfig / clearCache

```java
/**
 * 从缓存中移除指定序列的配置。
 * 配置更新后调用此方法，下次 nextVal 将从 DB 重新加载。
 */
void evictConfig(String seqName);

/**
 * 清除全部配置缓存。
 */
void clearCache();
```

#### stop

```java
/**
 * 停止序列服务，关闭配置缓存定时刷新调度器。
 * 由 Spring 容器在上下文关闭时调用（destroyMethod="stop"）。
 */
void stop();
```

#### getSequenceInfo

```java
/**
 * 获取指定序列的当前状态信息。
 * 用于监控管理和故障排查。
 *
 * @return SequenceInfo 包含配置信息、currVal、双缓冲状态
 */
SequenceInfo getSequenceInfo(String seqName);
```

### 5.2 SessionHolder 工具方法

`SessionHolder` 是会话级序列值持有者，基于 `ThreadLocal` 实现 Oracle CURRVAL 语义。

```java
// 设置当前会话的序列值（由 SequenceService.nextVal 自动调用）
SessionHolder.setCurrVal(seqName, value);

// 获取当前会话的序列值
SessionHolder.getCurrVal(seqName);  // 未初始化则抛出 CURRVAL_NOT_INITIALIZED

// 清理当前会话的所有序列值（HTTP 请求结束后自动清理）
SessionHolder.clear();
```

#### wrap 方法（非 HTTP 场景）

```java
// 包装 Runnable，执行后自动清理 ThreadLocal
SessionHolder.wrap(() -> {
    long val = sequenceService.nextVal("SEQ_NAME");
    long curr = sequenceService.currVal("SEQ_NAME");
    // 业务逻辑...
});

// 包装 Callable，执行后自动清理 ThreadLocal
Future<Long> future = executor.submit(SessionHolder.wrap(() -> {
    return sequenceService.nextVal("SEQ_NAME");
}));
```

### 5.3 创建序列配置（Builder）

```java
import com.ccb.jx.seq.model.Mode;
import com.ccb.jx.seq.model.SequenceConfig;

// 方式一：使用 Builder 创建自定义配置
SequenceConfig config = SequenceConfig.builder("ORDER_NO")
    .mode(Mode.CACHED)
    .step(1000)
    .startWith(10000L)
    .minValue(1L)
    .maxValue(999999999L)
    .cycle(0)
    .incrementBy(1)
    .description("订单号序列")
    .build();
```

## 6. CURRVAL 会话隔离

### 6.1 Oracle 语义说明

`currVal(seqName)` 返回**当前会话（线程）**最后一次成功调用 `nextVal(seqName)` 的值：

- 不同会话的 `currVal` 返回值相互独立
- 首次调用 `currVal` 前必须先调用 `nextVal`，否则抛出 `CURRVAL_NOT_INITIALIZED` 异常
- 与 Oracle Sequence 的 `CURRVAL` 伪列行为完全一致

### 6.2 HTTP 自动清理机制

在 Spring Boot Web 环境中，`SessionHolderCleanupInterceptor` 作为 Spring MVC `HandlerInterceptor` 自动注册（拦截所有 `/**` 路径），在 `afterCompletion` 阶段自动调用 `SessionHolder.clear()`。

**无需任何额外配置**——每个 HTTP 请求完成后，ThreadLocal 自动清理，防止内存泄漏。

### 6.3 非 HTTP 场景用法

在**定时任务**、**消息队列消费者**、**手动线程池**等非 HTTP 场景中，ThreadLocal 不会被自动清理。您有以下选择：

#### 方案一：手动 try-finally

```java
@Component
public class ScheduledTask {

    @Autowired
    private SequenceService sequenceService;

    @Scheduled(cron = "0 0 0 * * ?")  // 每天零点执行
    public void dailyReset() {
        try {
            long val = sequenceService.nextVal("DAILY_SEQ");
            // 业务逻辑...
        } finally {
            SessionHolder.clear();
        }
    }
}
```

#### 方案二：使用 wrap() 包装（推荐）

```java
// 定时任务
@Scheduled(cron = "0 0/5 * * * ?")
public void processMessages() {
    SessionHolder.wrap(() -> {
        long msgId = sequenceService.nextVal("MSG_ID");
        // 业务逻辑...
        System.out.println("消息ID: " + msgId);
    }).run();
}

// 消息队列消费者
messageListener.onMessage(message -> {
    SessionHolder.wrap(() -> {
        long id = sequenceService.nextVal("MSG_SEQ");
        // 处理消息...
    }).run();
});

// 线程池提交
executor.submit(SessionHolder.wrap(() -> {
    long id = sequenceService.nextVal("ASYNC_SEQ");
    long curr = sequenceService.currVal("ASYNC_SEQ");
    // 异步业务逻辑...
    return curr;
}));
```

## 7. 配置缓存与刷新

### 7.1 缓存机制

`SequenceService` 使用 `ConcurrentHashMap` 缓存序列配置，首次访问 `getConfig(seqName)` 时懒加载：

```
sequence_config 表  ── 首次访问 ──→  ConcurrentHashMap 缓存  ── 下次访问 ──→ 直接返回缓存
                                       ↑
                                       每隔 configCacheTtlMinutes 清空
```

### 7.2 刷新逻辑

`ConfigCacheRefresher` 由 `ScheduledExecutorService`（daemon 线程）定时调度，间隔由 `sequence.config-cache-ttl-minutes` 控制（默认 5 分钟）。

每次刷新执行三步操作：

1. **`cleanupStaleBuffers()`** — 查询 DB 全量配置，移除已从配置表中删除的序列的 DoubleBuffer
2. **`refreshConfigCache()`** — 清空配置缓存，下次访问时从 DB 重新加载
3. **`cleanupOldRecords()`** — 删除 `sequence_record` 中 `alloc_time` 超过 30 天的旧记录

### 7.3 对 CACHED 模式的影响

缓存刷新**不会**影响正在使用的 DoubleBuffer：

- 已加载的号段在内存中继续分配，不受缓存清空影响
- 只有 `getConfig()` 的配置读取受缓存影响
- DoubleBuffer 的清理仅在 `cleanupStaleBuffers()` 中针对已删除的序列执行

### 7.4 懒失效策略

配置缓存采用**懒失效**策略——缓存被清空后，不立即重新加载，而是在下次 `getConfig()` 调用时按需加载。这意味着：

- DBA 修改配置后，最多在 `configCacheTtlMinutes` 时间内生效
- 如果需要立即生效，可调用 `sequenceService.evictConfig(seqName)` 或 `sequenceService.clearCache()`
- 失败重试：连续失败 3 次以上升级为 ERROR 日志

## 8. 懒创建行为

### 8.1 首次 nextVal 自动创建

当 `sequence_config` 表中不存在对应 `seq_name` 时，`SequenceService.getConfig()` 在第一次访问时自动执行：

1. 调用 `INSERT IGNORE` 插入默认配置行
2. 重新查询获取实际写入的行
3. 写入缓存并返回

**默认配置**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| mode | STRICT | 默认严格连续模式 |
| step | 1000 | 号段步长（CACHED 生效） |
| increment_by | 1 | 序列步长（STRICT 生效） |
| min_value | 1 | 最小值 |
| max_value | `Long.MAX_VALUE` (9223372036854775807) | 最大值 |
| cycle | 0 | 不循环 |
| start_with | 1 | 起始值 |

### 8.2 start_with 修正

当 `max_id` 为 DDL 默认值（1）且 `start_with > 1` 时，首次 `nextVal` 会进行自动修正：

```sql
-- 假设 sequence_config: max_id=1, start_with=10000, increment_by=1
-- 自动修正为：currentMaxId = 10000 - 1 = 9999
-- 首次 nextVal 返回：9999 + 1 = 10000 ✅
```

### 8.3 从 1 开始生成序列

默认 `start_with=1`，无需任何额外配置。首次 `nextVal("MY_SEQ")` 返回 1。

```java
long first = sequenceService.nextVal("MY_SEQ");  // 返回 1
long second = sequenceService.nextVal("MY_SEQ");  // 返回 2
```

### 8.4 并发安全

多个实例同时首次调用同一序列名时，`INSERT IGNORE` 保证**仅一条写入生效**，其余实例忽略冲突，重新查询获取实际写入的行。

## 10. 优雅关闭

### 10.1 流程概述

`SequenceShutdownHook` 提供双入口保障的优雅关闭机制：

| 触发源 | 说明 |
|--------|------|
| `ApplicationListener<ContextClosedEvent>` | Spring 容器正常关闭时触发（主要入口） |
| `@PreDestroy` | Bean 销毁回调（兜底入口） |

`AtomicBoolean` CAS 操作保证无论触发几次，关闭逻辑**仅执行一次**。

### 10.2 关闭步骤

```
Step 1: ServiceState RUNNING → SHUTTING_DOWN
        └── 新 nextVal() 请求快速失败，抛出 SHUTTING_DOWN 异常

Step 2: 关闭异步预加载线程池
        ├── executor.shutdown()
        ├── awaitTermination(5s)
        └── 超时则 shutdownNow() 强制终止

Step 3: persistUnusedSegments()
        └── 遍历 bufferMap，记录未消耗号段范围到日志
            格式: seqName, current, end, remaining

Step 4: ServiceState SHUTTING_DOWN → SHUTDOWN
```

### 10.3 资源释放策略

| 资源 | 释放方式 | 超时 |
|------|----------|------|
| 异步预加载线程池 | shutdown → awaitTermination → shutdownNow | 5s |
| 配置缓存调度器 | shutdown → awaitTermination → shutdownNow | 3s |
| 独立 HikariCP 连接池 | Spring 自动推断并调用 close() | — |
| 业务数据源连接池 | SharedDataSourceWrapper 不实现 Closeable，不关闭 | — |
| 内存号段 | 仅记录日志，不回写数据库 | — |

## 12. 常见问题 FAQ

### Q1: step 在 STRICT 模式下有意义吗？

**没有意义。** `step` 字段仅在 CACHED 模式下控制每次从数据库预取的号段大小。STRICT 模式下，实际控制每次递增步幅的是 `increment_by` 字段。默认创建的 STRICT 序列 `step=1000` 不生效，只是一个占位值。

### Q2: 首次 nextVal 为什么返回 2 不是 1？

**因为 `start_with` 未正确修正。** 检查两种情况：

- 如果 `max_id` 为 DDL 默认值（1），且 `start_with=1`（也是默认值），修正逻辑判断 `start_with > 1` 为 false，不修正——首次 `nextVal` 返回 `1 + increment_by = 2`。
- 如需从 1 开始序列，请让 `start_with=1` 并使用 `increment_by=1`，首次调用返回 1。
- 如需从 10000 开始，设 `start_with=10000, increment_by=1`，首次返回 10000。

> **规则**：当 `max_id=1`（DDL 默认）且 `start_with > 1` 时，自动修正 `currentMaxId = start_with - increment_by`，使首次返回 `start_with`。

### Q3: 如何让序列从 1 开始生成？

默认配置 `start_with=1`，`increment_by=1`，首次调用 `nextVal("SEQ")` 即返回 1：

```java
long val = sequenceService.nextVal("SEQ");  // 返回 1
```

### Q4: 多实例下 STRICT 模式能保证不冲突吗？

**能。** STRICT 模式通过 `SELECT ... FOR UPDATE` 行级悲观锁保证多实例串行化：

- 实例 A 执行 `nextVal` → 获取行锁 → 读取/更新/提交 → 释放锁
- 实例 B 等待行锁释放 → 获取行锁 → 读取到最新值 → 更新/提交
- 任何时刻只有一个实例能操作序列行，保证无冲突

### Q5: CACHED 模式崩溃后会丢失号段吗？

**会。** 这是预期行为（Oracle 兼容）：

- 实例宕机时，当前 Segment 中未消耗完的序列号**永久丢失**
- 这些值不会被回收或回填（已分配出的号段可能已被业务使用）
- 这种现象被称为"gap"，是 Oracle Sequence 在 CACHE 模式下的标准行为
- 优雅关闭时，未消耗的号段范围会被记录到日志，但不会回写数据库

### Q6: NATIVE 模式的懒创建自愈如何工作？

两层懒创建：

1. **Level 1** — `sequence_config` 行不存在时，`SequenceService.getConfig()` 自动 `INSERT IGNORE` 默认配置（mode=STRICT）
2. **Level 2** — DB Sequence 对象不存在时，`NativeStrictSequence.nextVal()` 捕获异常，自动执行 `CREATE TDSQL_SEQUENCE` 并重试 `nextVal()`

此外，`NativeSequenceSynchronizer` 在应用启动时扫描所有 `mode=STRICT` 的序列，为缺失 DB Sequence 的序列执行创建操作。

### Q7: 未配置独立数据源时怎么办？

自动回退（Fallback）到应用的主数据源：

```yaml
# 不配置 sequence.datasource.*
# 组件自动复用 spring.datasource.* 定义的数据源
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/business_db
    username: root
    password: root123
```

回退模式使用 `SharedDataSourceWrapper` 包装业务数据源，确保 Spring 容器关闭时**不会**误关闭业务连接池。

### Q8: sequence_record 在哪种模式下使用？

**仅 CACHED 模式**使用 `sequence_record` 表。每种类型各一次 INSERT（非 UPDATE）：

| 模式 | 是否 INSERT | 说明 |
|------|-------------|------|
| STRICT | **否** | 直接 FOR UPDATE + UPDATE max_id，不记录审计 |
| CACHED | **是** | 每次加载新号段时 INSERT 一条 ALLOCATED 记录 |
| NATIVE | **否** | 仅 best-effort 更新 max_id，不记录审计 |

30 天前的旧记录由 `ConfigCacheRefresher.cleanupOldRecords()` 定期清理。

### Q9: CACHED 模式下 increment_by 是否生效？

**不生效。** CACHED 模式下连续值之间的步长始终为 1（`AtomicLong.getAndIncrement()`）。`increment_by` 仅在 STRICT 模式下生效。

### Q10: 业务事务回滚后，序列号会回退吗？

**不会。** Oracle 兼容语义：序列号一旦生成即不可回退。

- STRICT：`PROPAGATION_REQUIRES_NEW` 独立事务，序列值提交后与业务事务解耦
- CACHED：纯内存分配，与业务事务无关
- 业务回滚只影响业务数据，不回收已生成的序列号

## 13. 运维操作

### 13.1 新增序列

#### 方式一：自动懒创建（推荐）

直接调用 `nextVal("NEW_SEQ")`，组件自动插入默认配置（mode=STRICT, start_with=1）。

```java
long val = sequenceService.nextVal("NEW_SEQ");  // 自动创建默认配置
```

#### 方式二：手动 INSERT（自定义参数）

```sql
INSERT INTO sequence_config (seq_name, mode, step, increment_by, start_with, min_value, max_value, cycle, description)
VALUES ('ORDER_NO', 'CACHED', 1000, 1, 10000, 1, 999999999, 0, '订单号序列');
```

### 13.2 修改序列配置

直接 `UPDATE sequence_config` 表：

```sql
UPDATE sequence_config
SET increment_by = 5,          -- 修改步幅
    max_value = 999999999,     -- 修改最大值
    cycle = 1                  -- 开启循环
WHERE seq_name = 'ORDER_NO';
```

**配置传播延迟**：最多 `configCacheTtlMinutes`（默认 5 分钟）后生效。如需立即生效：

```java
// 方式一：清除单个缓存
sequenceService.evictConfig("ORDER_NO");

// 方式二：清除全部缓存
sequenceService.clearCache();
```

> **注意**：修改 `mode`（STRICT ↔ CACHED）需谨慎，运行时切换模式可能导致已分配的号段出现预期之外的行为。

### 13.3 删除序列

```sql
DELETE FROM sequence_config WHERE seq_name = 'OBSOLETE_SEQ';
```

下次 `ConfigCacheRefresher` 运行时，会清理对应的 DoubleBuffer 和配置缓存。

> **安全警告**：删除序列前请确认没有任何业务代码仍在调用此序列，否则将抛出 `SEQ_NOT_FOUND` 异常。

### 13.4 动态步长调优

CACHED 模式的动态步长会自动调整，一般情况下无需人工干预。可以通过监控接口观察步长变化：

```
GET /sequence/list
```

如果数据库更新周期偏离目标（15 分钟）较多，可考虑：

- **更新周期过短**（< 5 分钟）：说明序列消耗过快，动态步长会自动增大，但受限于阻尼（单次 ≤ 2 倍），可能需要多个调整周期才能稳定
- **更新周期过长**（> 60 分钟）：说明序列消耗过慢，步长偏大，动态步长会自动缩小
- **极端情况**：如果步长调整不满足业务要求，可直接 `UPDATE sequence_config SET step = N WHERE seq_name = '...'`

### 13.5 配置变更传播延迟

| 变更类型 | 生效时间 | 说明 |
|----------|----------|------|
| `mode` 变更 | 缓存刷新后 | 下次访问配置时生效（最多 5 分钟） |
| `step` 变更 | 下次号段加载 | CACHED 模式下次加载新号段时使用新步长 |
| `increment_by` 变更 | 缓存刷新后 | STRICT 模式下生效 |
| `start_with` 变更 | 仅新建序列 | 已有序列的 start_with 已生效，修改不影响已生成的值 |
| `max_value` 变更 | 缓存刷新后 | 下次 nextVal 检测到新边界 |
| `cycle` 变更 | 缓存刷新后 | 下次 nextVal 使用新循环策略 |
| 新建序列 | 立即 | 懒创建或手动 INSERT 后即可使用 |
| 删除序列 | 缓存刷新后 | 缓存中的配置在 TTL 后才移除 |
