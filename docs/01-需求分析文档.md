## 1. 项目背景

### 1.1 问题陈述

Oracle 数据库提供原生的 Sequence 对象，支持通过 `NEXTVAL` 和 `CURRVAL` 获取严格连续的序列号，广泛应用于业务流水号、订单编号、单据序号等场景。MySQL 及国产 TDSQL（MySQL 兼容分支）未提供等价的 Sequence 机制，仅支持表级 `AUTO_INCREMENT`，无法满足以下需求：

- 多表共享同一序列
- 自定义步长、边界值、循环策略
- 跨实例严格连续递增

### 1.2 参考方案

美团 Leaf 是业界成熟的分布式 ID 生成方案，其核心设计包括：

- **号段模式（Segment）**：批量预取号段到内存，减少数据库访问
- **双 Buffer 机制**：当前号段与备用号段交替，实现零等待切换
- **动态步长调整**：根据消耗速率自动调整步长，稳定数据库更新周期

本组件在借鉴 Leaf 核心思想的基础上，扩展支持 Oracle 风格的序列语义（严格连续、循环控制、自定义步长等），并新增 TDSQL 原生 Sequence 支持。号段缓存采用自定义双 Buffer 机制实现，无需引入外部缓存组件。

### 1.3 业务目标

| 目标编号 | 目标描述 | 验收标准 |
|----------|----------|----------|
| BG-01 | 为 TDSQL + Spring Boot 环境提供 Oracle 兼容的 Sequence 能力 | 功能测试通过率 100% |
| BG-02 | 支持严格连续、号段缓存、原生序列三种模式，供业务按需选择 | 严格连续模式 TPS ≥ 100；号段缓存模式 TPS ≥ 5000；原生序列模式 TPS 取决于数据库 |
| BG-03 | 数据库短时间不可用时，号段缓存模式仍能继续服务 | 模拟 DB 断开 5 分钟，服务不中断 |
| BG-04 | 支持 TDSQL 原生 Sequence，消除应用层行锁竞争 | 配置 `sequence.native-sequence-dialect=TDSQL` 后 STRICT 模式走原生序列 |

### 3.2 功能详细说明

#### FR-02 / FR-04：严格连续模式（STRICT）

```text-graph
┌─────────────────────────────────────────┐
│           严格连续模式时序图              │
├─────────────────────────────────────────┤
│                                         │
│  应用实例A          应用实例B          TDSQL │
│     │                  │                │ │
│     │ ── nextVal() ──► │                │ │
│     │                  │── BEGIN TX ───►│ │
│     │                  │── SELECT FOR   │ │
│     │                  │   UPDATE       │ │
│     │                  │◄── 返回 100   ─┤ │
│     │                  │                │ │
│     │ ◄── 等待 ────────│                │ │
│     │                  │── UPDATE 101 ─►│ │
│     │                  │── COMMIT ────► │ │
│     │                  │◄── 完成 ───────┤ │
│     │                  │                │ │
│     │── nextVal() ───► │                │ │
│     │                  │── BEGIN TX ───►│ │
│     │                  │── SELECT FOR   │ │
│     │                  │   UPDATE       │ │
│     │                  │◄── 返回 101   ─┤ │
│     │                  │── UPDATE 102 ─►│ │
│     │                  │── COMMIT ────► │ │
│     │◄── 返回 102 ─────│                │ │
│     │                  │                │ │
└─────────────────────────────────────────┘
```

**核心机制**：
- 通过 `SELECT ... FOR UPDATE` 行级悲观锁实现多实例串行化
- 事务内完成"读取当前值 → 计算下一个值 → 更新数据库"三步操作
- 锁随事务提交释放，下一个实例才能获取锁
- 使用 `TransactionTemplate`（`PROPAGATION_REQUIRES_NEW`, timeout=3s）编程式事务，解决自调用 `@Transactional` 失效问题
- 支持 `start_with` 修正：当 `max_id` 为 DDL 默认值（1）且 `start_with > 1` 时自动修正
- `increment_by` 在 STRICT 模式下生效，控制每次递增的步幅

#### FR-05 / FR-06：号段缓存模式（双 Buffer）

```text-graph

│      │                                  │
│      ▼                                  │
│   ┌─────────────┐                       │
│   │  加载 Segment0  │◄── 当前使用缓冲区    │
│   │  [1, 1000]   │                       │
│   └──────┬──────┘                       │
│          │                              │
│          ▼ 双条件触发预加载               │
│          │ 条件①: 使用率 ≥ 90%           │
│          │ 条件②: 剩余 < max(50, 30%)    │
│          │ （OR 逻辑，任一满足即触发）     │
│   ┌─────────────┐   ┌─────────────┐    │
│   │  Segment0    │   │  Segment1    │    │
│   │  [1, 1000]   │   │  [1001, 2000]│◄───┤── 异步预加载
│   │  当前使用    │   │  备用        │    │
│   └──────┬──────┘   └─────────────┘    │
│          │                              │
│          ▼ 用完 1000                    │
│   ┌─────────────┐   ┌─────────────┐    │
│   │  Segment0    │   │  Segment1    │    │
│   │  [空]        │   │  [1001, 2000]│◄───┤── 切换为当前
│   │  待加载      │   │  当前使用    │    │
│   └──────┬──────┘   └──────┬──────┘    │
│          │                  │           │
│          └────── 循环 ──────┘           │
│                                         │
└─────────────────────────────────────────┘
```

**核心机制**：
- 内存中维护两个 Segment 对象（当前使用 + 备用），通过自定义双 Buffer 机制管理
- 双条件预加载触发（OR 逻辑）：条件①使用率 ≥ 90%；条件②剩余数量 < `max(50, 总大小 × 0.3)`
- `preloadTriggered`（`AtomicBoolean`）CAS 保证仅触发一次异步加载
- `staleResetCount`（`AtomicInteger`）监控异步任务被线程池丢弃后的标记重置次数
- `asyncTaskSubmitter`（`volatile Runnable`）可插拔的异步任务提交器，未设置时降级为同步加载
- Segment 切换时无等待，TP999 延迟 < 1ms
- `next()` 方法最大重试 3 次（`MAX_RETRY = 3`），防止无限循环
- 无需引入 Caffeine 等外部缓存组件，降低依赖复杂度

**线程池配置**：
- 核心 20 线程，最大 100 线程
- `LinkedBlockingQueue(200)`
- `CallerRunsPolicy` 拒绝策略
- keepAlive = 60s

**Segment 模型**：
- 使用 `AtomicLong.getAndIncrement()` 实现无锁分配（step=1 始终）
- CACHED 模式下 `increment_by` 不生效，连续值之间步长始终为 1
- `next()` 返回 -1 表示号段耗尽
- `EmptySegment`（Null Object 模式）替代 null 引用，所有方法返回耗尽语义

#### FR-03：currVal 会话语义

**核心机制**：
- `currVal(seqName)` 返回当前会话最后一次成功调用 `nextVal(seqName)` 的值
- 当前会话首次调用 `currVal` 前必须先调用 `nextVal`，否则抛出 `CURRVAL_NOT_INITIALIZED` 异常
- 会话级隔离：不同会话的 `currVal` 返回值相互独立
- 此语义与 Oracle Sequence 的 `CURRVAL` 伪列行为一致

**会话定义**：
- 在 Spring Boot 环境中，一次 HTTP 请求的生命周期视为一个会话
- 会话内通过 `ThreadLocal<Map<String, Long>>` 维护每个序列的最后一次 nextVal 值
- `SessionHolder` 提供 `wrap(Runnable/Callable)` 方法，用于非 HTTP 线程场景的自动清理
- `SessionHolderCleanupInterceptor`（Spring MVC `HandlerInterceptor`）在 `afterCompletion` 阶段自动清理 ThreadLocal

#### FR-13：原生序列模式（NATIVE）

**核心机制**：
- 当配置 `sequence.native-sequence-dialect=TDSQL` 时，STRICT 模式的策略实现从 `StrictSequence`（FOR UPDATE）替换为 `NativeStrictSequence`（委托数据库原生 Sequence）
- `NativeSequenceDialect` 接口抽象不同数据库的原生 Sequence 差异，提供 `nextVal`、`createSequence`、`dropSequence`、`sequenceExists`、`isSequenceNotExist` 五个操作
- `TdsqlSequenceDialect` 使用 TDSQL 特有语法：`SELECT tdsql_nextval(?)`、`CREATE TDSQL_SEQUENCE`、`DROP TDSQL_SEQUENCE`、`SELECT tdsql_lastval(?)`
- `NativeStrictSequence` 的 `max_id` 更新采用 best-effort 策略：更新失败仅打 WARN 日志，不阻塞返回值

**两层懒创建**：
1. **Level 1** — `sequence_config` 行：由 `SequenceService.getConfig()` 在配置不存在时自动 `INSERT IGNORE` 创建默认配置
2. **Level 2** — DB Sequence 对象：由 `NativeStrictSequence.nextVal()` 在 `dialect.nextVal()` 抛出"Sequence 不存在"异常时，自动调用 `dialect.createSequence()` 并重试

**启动同步**：
- `NativeSequenceSynchronizer` 在应用启动时扫描所有 `mode=STRICT` 的序列，为每个未创建原生 Sequence 的序列执行 `CREATE`

#### FR-14：懒创建机制

**核心机制**：
- `SequenceService.getConfig(seqName)` 使用 `ConcurrentHashMap.computeIfAbsent` 实现缓存 + 懒加载
- 当 `sequence_config` 表中不存在对应行时，自动按默认参数 `INSERT IGNORE` 一条新记录
- 默认配置：`mode=STRICT`，`step=1000`，`increment_by=1`，`min_value=1`，`max_value=Long.MAX_VALUE`，`cycle=0`，`start_with=1`
- `INSERT IGNORE` 保证并发安全：多个实例同时首次调用同一序列名，仅一个实例实际写入，其余忽略
- 写入后重新查询获取实际行（并发场景下可能已被其他实例写入不同配置）

#### FR-15：数据源回退

**核心机制**：
- 当 `sequence.datasource.url` 和 `sequence.datasource.jdbc-url` 均未配置时，自动回退到应用自带的 `DataSource`
- 使用 `SharedDataSourceWrapper`（继承 `AbstractDataSource`，不实现 `Closeable`）包装应用数据源
- 防止 Spring 容器关闭时误关闭应用数据源（独立 HikariCP 数据源实现了 `Closeable`，Spring 会自动推断并调用 `close()`）
- 独立数据源模式下使用 HikariCP 连接池：maxPoolSize=5, minIdle=1, connectionTimeout=3000ms, maxLifetime=1800000ms

#### FR-17：策略模式路由

**核心机制**：
- `SequenceStrategy`（`@FunctionalInterface`）定义统一接口：`long nextVal(String seqName, SequenceConfig config)`
- `SequenceService` 通过 `Map<Mode, SequenceStrategy> strategyMap` 实现模式路由
- 三种实现：`StrictSequence`（FOR UPDATE）、`NativeStrictSequence`（原生 Sequence）、`CachedSequence`（号段缓存）
- `SequenceAutoConfiguration` 根据配置条件装配不同的 STRICT 策略实现：
  - `sequence.native-sequence-dialect=NONE`（默认）→ `StrictSequence`
  - `sequence.native-sequence-dialect=TDSQL` → `NativeStrictSequence`

#### FR-18：服务状态机

```text-graph
┌─────────────────────────────────────────┐
│          ServiceState 状态机             │
├─────────────────────────────────────────┤
│                                         │
│   ┌──────────┐    canTransitionTo    ┌──────────────┐    canTransitionTo    ┌─────────┐
│   │ RUNNING  │ ───────────────────► │ SHUTTING_DOWN │ ───────────────────► │ SHUTDOWN│
│   │ (接受请求)│                       │ (拒绝新请求)  │                       │ (已关闭) │
│   └──────────┘                       └──────────────┘                       └─────────┘
│                                         │
│                                         │ acceptRequests() = false
│                                         │ 抛出 SHUTTING_DOWN 异常
│                                         │
└─────────────────────────────────────────┘
```

**核心机制**：
- `ServiceState` 枚举定义三种状态：`RUNNING`、`SHUTTING_DOWN`、`SHUTDOWN`
- 状态转换单向不可逆：`RUNNING → SHUTTING_DOWN → SHUTDOWN`
- `canTransitionTo()` 方法校验转换合法性
- `acceptRequests()` 方法仅在 `RUNNING` 状态下返回 true
- `CachedSequence` 持有 `volatile ServiceState state` 字段，`nextVal()` 入口检查状态

#### FR-19：配置缓存刷新

**核心机制**：
- `ConfigCacheRefresher` 实现 `Runnable`，由 `SequenceService` 的 `ScheduledExecutorService`（daemon 线程）定时调度
- 调度间隔由 `sequence.config-cache-ttl-minutes` 控制，默认 5 分钟
- 三步操作：
  1. `cleanupStaleBuffers()` — 查询数据库全量配置，移除已从配置表中删除的序列的 DoubleBuffer
  2. `refreshConfigCache()` — 粗粒度清空整个配置缓存，确保 DBA 修改配置后能在 TTL 时间内生效
  3. `cleanupOldRecords()` — 删除 `sequence_record` 表中 `alloc_time` 超过 30 天的记录
- `refreshFailCount`（`AtomicInteger`）连续失败计数器，3 次以上升级为 ERROR 日志
- `SequenceService.stop()` 方法（`destroyMethod="stop"`）关闭调度器，3s 优雅等待

#### FR-20：空号段占位

**核心机制**：
- `EmptySegment` 继承 `Segment`，采用 Null Object 模式
- 全局唯一实例 `EmptySegment.INSTANCE`
- 所有方法返回耗尽语义：`next()=-1`、`remaining()=0`、`isExhausted()=true`、`usage()=1.0`、`isInitialized()=false`
- `DoubleBuffer` 初始化时 `current` 和 `standby` 均设为 `EmptySegment.INSTANCE`
- 消除 DoubleBuffer 中的 null 检查，简化代码逻辑

### 6.2 数据字典

| 数据项 | 类型 | 取值范围 | 业务含义 |
|--------|------|----------|----------|
| seq_name | VARCHAR(64) | 唯一，非空 | 序列的业务标识 |
| max_id | BIGINT | ≥ 1 | 已分配号段的最大值，下次分配从此值+1开始 |
| step | INT | [100, 1000000] | 每次从数据库批量获取的号段长度（CACHED 模式） |
| increment_by | INT | ≥ 1 | 序列每次递增的步幅（STRICT 模式） |
| mode | VARCHAR(16) | {STRICT, CACHED} | STRICT=严格连续模式；CACHED=号段缓存模式。NATIVE 模式通过 `sequence.native-sequence-dialect` 配置激活，不在 mode 列中体现 |
| start_with | BIGINT | ≥ 1 | 序列的起始生成值 |
| cycle | TINYINT | {0, 1} | 0=到达最大值报错；1=到达最大值循环到最小值 |
| status | VARCHAR(16) | {ALLOCATED, EXPIRED, RECYCLED} | 号段分配记录状态 |

### 6.3 持久化标记说明

- `sequence_record` 仅在 **CACHED 模式** 下使用（STRICT 模式不插入记录）
- 仅执行 INSERT 操作（ALLOCATED 状态），不执行 UPDATE 变更状态
- 30 天清理由 `ConfigCacheRefresher.cleanupOldRecords()` 执行，删除 `alloc_time` 超过 30 天的记录
- 号段加载与记录插入在同一事务中执行（`TransactionTemplate` + `REQUIRES_NEW`）

## 7. 约束条件

### 7.1 业务规则约束

| 约束编号 | 约束描述 | 影响范围 |
|----------|----------|----------|
| BR-01 | 同一序列名称全局唯一 | sequence_config.seq_name |
| BR-02 | 严格连续模式下，序列号必须严格递增（1,2,3...） | 严格连续模式 |
| BR-03 | 号段缓存模式下，同一实例内连续，实例间可能跳号 | 号段缓存模式 |
| BR-04 | 循环模式下，到达 max_value 后回绕到 min_value | cycle=1 |
| BR-05 | 非循环模式下，到达 max_value 后抛出异常 | cycle=0 |
| BR-06 | CACHED 模式下 `increment_by` 不生效，连续值步长始终为 1 | 号段缓存模式 |
| BR-07 | NATIVE 模式通过 `sequence.native-sequence-dialect` 配置激活，不在 `mode` 列中体现 | 原生序列模式 |

### 7.2 技术约束

| 约束编号 | 约束描述 | 说明 |
|----------|----------|------|
| TC-01 | 数据库事务隔离级别 ≥ READ COMMITTED | 保证 FOR UPDATE 锁生效 |
| TC-02 | 序列表必须为单表（全局表） | 避免分库分表导致分布式事务 |
| TC-03 | 应用实例时钟偏差 < 1 秒 | 影响监控统计，不影响序列正确性 |
| TC-04 | 内存中号段缓存随应用重启丢失 | 业务需接受号段缓存模式的跳号风险 |
| TC-05 | 使用 `TransactionTemplate` 编程式事务而非 `@Transactional` | 自调用场景下 Spring AOP 代理不拦截，`@Transactional` 失效 |

### 7.3 性能约束

| 约束编号 | 约束描述 | 阈值 |
|----------|----------|------|
| PC-01 | 严格连续模式下单实例 TPS 上限 | ~50 TPS（受行锁竞争限制） |
| PC-02 | 严格连续模式下 4 实例 TPS 上限 | ~100-200 TPS |
| PC-03 | 号段缓存模式下单实例 TPS 上限 | ~50000+ TPS（内存分配） |
| PC-04 | 数据库号段更新频率 | 每 15-30 分钟一次（动态调整） |
| PC-05 | 原生序列模式 TPS | 取决于数据库原生 Sequence 性能 |

## 11. 宕机抵御机制设计

### 11.1 设计原则

本组件遵循 Oracle 兼容的宕机处理原则：**gap 是预期行为，不尝试回收或回填**。

- CACHED 模式下，实例宕机时当前 Segment 剩余值永久丢失（gap），这是 Oracle Sequence 在 CACHE 模式下的标准行为
- STRICT 模式下，事务未提交的值丢失，最多产生 1 个空洞
- 不存在心跳机制、CrashRecoveryService 或 sequence_heartbeat 表
- 号段丢失不可恢复，`max_id` 不会回退

### 11.2 优雅关闭机制（Graceful Shutdown）

#### 11.2.1 触发条件

| 触发源 | 触发条件 | 说明 |
|--------|----------|------|
| `ContextClosedEvent` | Spring 容器正常关闭（K8s 缩容、滚动更新、正常停止） | 主要入口 |
| `@PreDestroy` | Bean 销毁回调 | 兜底入口 |

> **说明**：本组件作为 Spring Boot Starter jar 嵌入宿主应用，不直接处理 OS 级信号（SIGTERM/SIGINT）。信号由 JVM 和宿主应用处理，组件通过 Spring 生命周期事件感知关闭。两个入口通过 `AtomicBoolean` 保证关闭逻辑仅执行一次。

#### 11.2.2 执行流程

```text-graph
┌─────────────────────────────────────────┐
│           优雅关闭执行流程               │
├─────────────────────────────────────────┤
│                                         │
│  [ContextClosedEvent / @PreDestroy 触发] │
│    │                                    │
│    ▼                                    │
│  [AtomicBoolean CAS 保证仅执行一次]      │
│    │                                    │
│    ▼                                    │
│  [Step 1: 状态转换到 SHUTTING_DOWN]      │
│    │ 新 nextVal() 快速失败               │
│    │ 抛出 SHUTTING_DOWN 异常             │
│    ▼                                    │
│  [Step 2: 关闭异步预加载线程池]           │
│    │ executor.shutdown()                 │
│    │ awaitTermination(5s)                │
│    │ 超时 → shutdownNow()                │
│    ▼                                    │
│  [Step 3: 持久化未使用号段信息]           │
│    │ 遍历 bufferMap                      │
│    │ 记录未消耗号段范围到日志              │
│    │ 格式: seqName, current, end, remain │
│    ▼                                    │
│  [Step 4: 状态转换到 SHUTDOWN]           │
│    │                                    │
│    ▼                                    │
│  [结束]                                  │
│                                         │
└─────────────────────────────────────────┘
```

#### 11.2.3 资源释放策略

| 资源类型 | 释放策略 | 超时时间 |
|----------|----------|----------|
| 异步线程池 | `shutdown()` → `awaitTermination()` → `shutdownNow()` | 5s |
| 配置缓存调度器 | `shutdown()` → `awaitTermination()` → `shutdownNow()` | 3s |
| 独立数据库连接池 | HikariCP `close()`（Spring 自动推断） | - |
| 应用数据库连接池 | `SharedDataSourceWrapper` 不实现 `Closeable`，不关闭 | - |
| 内存号段 | 记录日志，不强制回写数据库 | - |

### 11.3 两种模式下的宕机处理对比

| 对比维度 | 严格连续模式 | 号段缓存模式 |
|----------|-------------|-------------|
| **核心诉求** | 绝对连续，不允许跳号 | 高性能，允许有限跳号 |
| **号段分配方式** | 实时从数据库获取，每次+increment_by | 批量预取到内存 |
| **宕机时丢失的号段** | 仅当前正在处理的 1 个号 | 当前 Segment 剩余全部号段 |
| **号段处理策略** | 直接丢弃，永不回收 | 直接丢弃，永不回收（gap 为预期行为） |
| **是否破坏连续性** | 可能产生 1 个空洞 | 可能产生大量空洞 |
| **号段浪费程度** | 极小（每次仅丢 1 个） | 较大（可能丢数千个） |

**不可回收的根本原因**：序列号一旦生成就可能被写入业务表、发送给外部系统或打印在纸质单据上。回收后重新分配可能导致同一序列号对应两个不同业务实体的严重数据错误。

#### 模式选择建议

| 业务场景 | 推荐模式 | 宕机后处理 | 号段浪费 |
|----------|----------|-----------|----------|
| 财务凭证号、发票号 | 严格连续 | 丢弃，不回收 | 极小 |
| 订单号（需严格连续） | 严格连续 | 丢弃，不回收 | 极小 |
| 分库分表主键 | 号段缓存 | 丢弃，接受 gap | 较大 |
| 日志ID、消息ID | 号段缓存 | 丢弃，接受 gap | 较大 |
| TDSQL 环境需严格连续 | 原生序列 | 数据库原生保证 | 无 |

## 12. 事务隔离设计

### 12.1 核心原则：序列事务必须与业务事务隔离

#### 12.1.1 问题场景

```text-graph
┌─────────────────────────────────────────────────────────────┐
│              错误设计：序列号与业务共用事务                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [创建订单业务]                                              │
│    │                                                        │
│    ├── BEGIN TX (业务事务)                                   │
│    │                                                        │
│    ├── ► 获取订单号 nextVal('ORDER_NO') ◄── 内部开启子事务?   │
│    │      ├── BEGIN TX (序列事务)                            │
│    │      ├── SELECT FOR UPDATE sequence                     │
│    │      ├── UPDATE sequence SET current=101               │
│    │      └── COMMIT (序列事务)                              │
│    │                                                        │
│    ├── INSERT INTO order (order_no=1001, ...)               │
│    ├── INSERT INTO order_item (...)                         │
│    ├── 扣减库存 (...)                                       │
│    │                                                        │
│    └── COMMIT (业务事务)                                     │
│                                                             │
│  问题:                                                       │
│  1. 序列事务已提交，业务事务可能回滚                          │
│     → 订单号 1001 已被占用，但订单不存在                      │
│     → 号段永久浪费                                           │
│                                                             │
│  2. 业务事务持有锁时间过长                                    │
│     → 序列锁被业务操作（库存扣减等）延长                      │
│     → 其他实例获取序列号阻塞                                  │
│     → TPS 急剧下降                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 12.1.2 正确设计：独立事务（REQUIRES_NEW）

```text-graph
┌─────────────────────────────────────────────────────────────┐
│              正确设计：序列号独立事务                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [创建订单业务]                                              │
│    │                                                        │
│    ├── ► 获取订单号 nextVal('ORDER_NO') ◄── 独立事务         │
│    │      ├── BEGIN TX (PROPAGATION_REQUIRES_NEW)           │
│    │      ├── SELECT FOR UPDATE sequence                     │
│    │      ├── UPDATE sequence SET current=101               │
│    │      └── COMMIT (序列事务立即提交)                       │
│    │                                                        │
│    ├── BEGIN TX (业务事务)                                   │
│    │                                                        │
│    ├── INSERT INTO order (order_no=1001, ...)               │
│    ├── INSERT INTO order_item (...)                         │
│    ├── 扣减库存 (...)                                       │
│    │                                                        │
│    ├── 业务异常?                                             │
│    │   ├── 是 → ROLLBACK (业务回滚)                          │
│    │   │       订单号 1001 已生成，但不回收                   │
│    │   └── 否 → COMMIT (业务提交)                            │
│    │                                                        │
│    └── 业务事务结束                                          │
│                                                             │
│  结果:                                                       │
│  • 序列号生成与业务完全解耦                                   │
│  • 序列锁持有时间极短（毫秒级）                               │
│  • 业务回滚不影响序列号（接受跳号）                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 12.2 三种模式的事务策略对比

| 对比维度 | 严格连续模式 | 号段缓存模式 | 原生序列模式 |
|----------|-------------|-------------|-------------|
| **事务传播行为** | `REQUIRES_NEW`（强制独立） | 内存分配无需事务；号段加载用 `REQUIRES_NEW` | 无应用级事务（依赖 DB 原生原子性） |
| **事务实现方式** | `TransactionTemplate` 编程式 | `TransactionTemplate` 编程式 | 无 |
| **事务超时** | 3 秒 | 3 秒 | 无 |
| **锁持有时间** | 仅序列操作期间（~5ms） | 号段加载时（~10ms），日常分配无锁 | 无应用层锁 |
| **业务回滚影响** | 序列号不回收（跳号 1 个） | 序列号不回收（跳号一个号段） | 序列号不回收 |
| **并发性能** | 100-200 TPS | 5000+ TPS | 取决于数据库 |
| **事务隔离级别** | `READ_COMMITTED` 即可 | `READ_COMMITTED` 即可 | 由数据库决定 |

### 12.3 关键设计决策

#### 12.3.1 为什么使用 TransactionTemplate 而非 @Transactional

```text-graph
┌─────────────────────────────────────────────────────────────┐
│     TransactionTemplate vs @Transactional 决策分析           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  @Transactional 的问题:                                      │
│                                                             │
│  SequenceService.nextVal()                                  │
│    └── this.nextValStrict(seqName)  ← 自调用               │
│          └── @Transactional ← Spring AOP 代理不拦截自调用!  │
│              → 事务不生效!                                   │
│                                                             │
│  TransactionTemplate 的优势:                                 │
│                                                             │
│  SequenceService.nextVal()                                  │
│    └── strictSequence.nextVal(seqName, config)              │
│          └── transactionTemplate.execute(status -> {...})   │
│              → 编程式事务，不依赖 AOP 代理                    │
│              → 无论调用路径如何，事务始终生效                  │
│                                                             │
│  结论: 使用 TransactionTemplate 保证事务可靠性               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 12.3.2 事务隔离决策清单

| # | 决策项 | 决策结果 | 理由 |
|---|--------|----------|------|
| 1 | 业务回滚后序列号是否回收？ | **不回收** | 简化设计，接受跳号（Oracle 语义） |
| 2 | 号段缓存模式下业务回滚导致号段内跳号？ | **不处理** | 号段内天然允许跳号 |
| 3 | 是否需要分布式事务（如 Seata）？ | **不需要** | 序列服务独立提交，与业务解耦 |
| 4 | 序列事务超时时间？ | **3 秒** | 防止死锁，快速失败 |
| 5 | 事务隔离级别？ | **READ_COMMITTED** | 满足需求，性能最优 |
| 6 | 事务实现方式？ | **TransactionTemplate** | 自调用场景下 @Transactional 失效 |

### 13.1 术语表
